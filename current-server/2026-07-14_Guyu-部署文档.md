## 3. 基础设施部署（Docker）

### 3.1 一键启动中间件

已在 `deploy/docker-compose.middleware.yml` 准备 Compose 文件。实际运行采用分开的 docker run 命令，推荐使用 Compose：

```bash
# 持久化目录
mkdir -p /data/guyu/{mysql,nacos,qdrant}

# MySQL 5.7
docker run -d --name guyu-mysql --restart unless-stopped \
  -e MYSQL_ROOT_PASSWORD=root \
  -e MYSQL_DATABASE=appdata \
  -e TZ=Asia/Shanghai \
  -p 3306:3306 \
  -v /data/guyu/mysql:/var/lib/mysql \
  mysql:5.7 \
  --character-set-server=utf8mb4 --collation-server=utf8mb4_unicode_ci --max_connections=500

# Redis (给 knowledge-server / gateway 用，无密码)
docker run -d --name guyu-redis --restart unless-stopped \
  -p 6379:6379 \
  redis:7 redis-server --appendonly yes

# Redis (给 im-server 用，需密码)
docker run -d --name guyu-redis-2 --restart unless-stopped \
  -p 6380:6379 \
  redis:7 redis-server --appendonly yes --requirepass redis

# Nacos (v2.3.2，单机模式，关闭认证)
docker run -d --name guyu-nacos --restart unless-stopped \
  -e MODE=standalone \
  -e NACOS_AUTH_ENABLE=false \
  -e JVM_XMS=512m -e JVM_XMX=512m -e JVM_XMN=256m \
  -p 8848:8848 -p 9848:9848 \
  -v /data/guyu/nacos:/home/nacos/data \
  nacos/nacos-server:v2.3.2

# Qdrant (v1.10.1 向量库)
docker run -d --name guyu-qdrant --restart unless-stopped \
  -p 6333:6333 -p 6334:6334 \
  -v /data/guyu/qdrant:/qdrant/storage \
  qdrant/qdrant:v1.10.1
```

或者使用 Compose 文件：

```bash
cd /data/guyu/source/guyu/deploy
docker compose -f docker-compose.middleware.yml up -d mysql redis nacos qdrant
```

> **注意:** Compose 文件中的 Nacos 启用了认证 (`NACOS_AUTH_ENABLE=true`)，与当前运行实例不同。如使用 Compose，需调整环境变量或用 `nacos / PaCl@PangHu$233!Ok` 登录。当前机器实际用了 `NACOS_AUTH_ENABLE=false`，所以 nacos/nacos 即可访问。

## 5. 前端构建

### 5.1 admin_web（知识管理后台）

> Vue 3 + Vite 5 + Vue Router 4  
> Node 版本要求: 18+（当前环境 Node 22 已验证）

**问题:** `package.json` 依赖 `wiki-front-drawio@^0.0.47`（太保内网私有 npm 包），外网拉不到。

**解决方法:** 移除 `wiki-front-drawio` 依赖 + 替换相关源码为 stub。

```bash
cd /data/guyu/source/guyu/admin_web

# 1. 备份原始 package.json
cp package.json package.json.bak

# 2. 临时移出 wiki-front-drawio
python3 -c "
import json
p = json.load(open('package.json'))
p['dependencies'].pop('wiki-front-drawio', None)
json.dump(p, open('package.json', 'w'), indent=2, ensure_ascii=False)
"

# 3. 安装依赖（用 npmmirror 镜像）
npm install --registry=https://registry.npmmirror.com --legacy-peer-deps

# 4. 恢复 package.json（保持构建信息准确）
cp package.json.bak package.json

# 5. 构建
npm run build
# 输出在 dist/ 目录

# 6. 部署到 nginx 目录
rm -rf /data/guyu/nginx/admin_web
cp -r dist /data/guyu/nginx/admin_web
```

**如果还是遇到包下载失败**（某些 antd 子包需要手动安装），参考 `ADMIN_WEB_DEPLOY.md` 的 `npm pack` 方案逐个下载。

| `src/views/Staff/components/tree.vue` | `watch(treeObject, ...)` → `watch(() => treeObject, ...)` |

| `src/views/GuYu/PersonSpace/Knowledge/Drawio/MxGraph.vue` | 替换为占位 stub（不依赖 wiki-front-drawio） |

### 7.2 Java 服务启动命令

所有 Java 服务都用 `nohup + java -jar` 后台运行。日志在 `/data/guyu/logs/`。

```bash

export LOG_DIR=/data/guyu/logs

# 公用环境变量
```

#### 7.2.1 knowledge-server (8089)

```bash
JAR=$(find "$BASE_DIR/knowledge/server/target" -maxdepth 2 -name '*.jar' ! -name '*sources*' ! -name '*javadoc*' | sort | tail -1)

nohup java -Xms1g -Xmx2g \
  -DenvType=dev \
  -DnacosServerAddr=127.0.0.1:8848 \
  -DnacosUserName=nacos \
  -DnacosPassWord=PaCl@PangHu$233!Ok \
  -DnacosNameSpace=public \
  -Dspring.datasource.url="jdbc:mysql://127.0.0.1:3306/appdata?useSSL=false&useUnicode=true&characterEncoding=utf-8&allowPublicKeyRetrieval=true" \
  -Dspring.datasource.username=root \
  -Dspring.datasource.password=root \
  -Dspring.redis.host=127.0.0.1 \
  -Dspring.redis.port=6379 \
  -jar "$JAR" \
```

## 10. 已知坑 + 排错指南

### 10.1 mybatis-plus 版本冲突 (IncompatibleClassChangeError)

**现象:** 启动 knowledge-server 或 im-platform 时抛 `IncompatibleClassChangeError`。

**原因:** `knowledge/pom.xml` 的 `dependencyManagement` 将 mybatis-plus 锁到 `3.3.2`，但 `im-platform/pom.xml` 明确依赖 `mybatis-plus 3.5.3.2`。两个版本混用时 JVM 加载了不兼容的 class。

**解决方案:**

```bash
# 方案: 统一 knowledge 父 POM 的 mybatis-plus 版本
# 编辑 /data/guyu/source/guyu/knowledge/pom.xml
# 将:
#   <mybatisplus-spring-boot-starter.version>3.3.2</mybatisplus-spring-boot-starter.version>
# 改为:
#   <mybatisplus-spring-boot-starter.version>3.5.3.2</mybatisplus-spring-boot-starter.version>
# 同时更新 dependencyManagement 中 mybatis-plus / mybatis-plus-core / mybatis-plus-extension / mybatis-plus-annotation 的版本也为 3.5.3.2
```

但是注意 `im-platform` 的 `mybatis-plus-boot-starter` 依赖排除了 `mybatis-plus`（见 pom.xml L59-L65），然后又显式引入了 3.5.3.2，这本身就有问题。

**当前环境的实际状态:** knowledge-server 已在运行（8089），使用的是本地 `application-dev.yml` 配置。但 im-server 未运行，怀疑也与 mybatis-plus 版本冲突有关。

### 10.2 pagehelper 版本冲突 (IllegalAccessError)

**现象:** 在 Java 8 环境下启动报 `IllegalAccessError`，涉及 `DefaultCountSqlParser` 构造方法。

**原因:** `pagehelper-spring-boot-starter 2.1.0` 依赖的 `pagehelper 6.1.0` 使用了 Java 9+ 的 `invokedynamic` 指令，Java 8 运行时无法解析。

**解决方案:**

在 `im/pom.xml` 和 `knowledge/pom.xml` 的 `dependencyManagement` 中锁定 pagehelper 版本到 5.x：

```xml
<dependency>
    <groupId>com.github.pagehelper</groupId>
    <artifactId>pagehelper</artifactId>
    <version>5.3.4</version>
</dependency>
```

或者在 `im-platform/pom.xml` 中显式声明低版本：

```xml
<dependency>
    <groupId>com.github.pagehelper</groupId>
    <artifactId>pagehelper-spring-boot-starter</artifactId>
    <version>1.4.6</version>  <!-- 这个版本依赖 pagehelper 5.x -->
</dependency>
```

### 10.3 CyberArk 禁用

**原因:** 源码中大量使用 `@Value("${cyberark.xxx}")` 从 CyberArk 获取凭据。在无 CyberArk 的环境中，启动会失败。

**当前处理方式:** 已用以下方式绕过：

1. **`gateway/application-dev.yml`** — 直接写死空值：
   ```yaml
   cyberark:
     url: ''
     appId: local
     appKey: local
     ...
   ```

2. **`knowledge/server/application-dev.yml`** — 保留了 CyberArk 配置但未使用（数据源直接写了 `root/root`）。

3. **Nacos 中的配置** — 将 CyberArk 对应的凭据直接以明文写入配置（如 MySQL 密码 `root`、Redis 密码 `redis`）。

**新机器上:** 确保所有 `application-dev.yml` 及 Nacos 配置中，数据源、Redis、UM 等凭据直接使用明文，不依赖 CyberArk。

### 10.4 wiki-front-drawio 私有 npm 包

### 12.1 knowledge-bank 编译后字节码错误 (NoSuchMethodError)

**问题现象:**

```
POST /admin-server/knowledge-bank/knowledgeBank/build
返回 500，server.log 中出现:
NoSuchMethodError: com.paic.csp.ai.knowledge.bank.utils.AuthUtils.addKnowledgeTeamMember
```

**根因:**

`guyu-biz-stubs` 模块中存在一个桩类 `StubXxx`，其 `addKnowledgeTeamMember(Object)` 方法签名与 `robot-workbench` 中的 `addKnowledgeTeamMember(Constant$Role)` 接口不一致。编译 `knowledge-bank` 时，旧的 `server.jar` 中该方法引用了错误的字节码描述符（`Object` 而非 `Constant$Role`），导致运行时 JVM 校验失败。

**解决方法:**

```bash
cd /data/guyu/source/guyu/knowledge/knowledge-bank
mvn clean package -DskipTests
# 将新编译的 jar 部署到:
# /data/guyu/source/guyu/knowledge/server/target/server.jar
```

> 关键：使用 `mvn clean package` 重新编译，而非直接替换 class 文件。Java 8 环境下用 javaagent 改 class 容易触发 JVM 内部 VerifyError，应用层重编译才是最稳的方案。

### 12.2 Python 向量服务地址配置错误

**问题现象:**

knowledge-server 日志中持续出现:
```
远程调用向量化失败
java.net.SocketTimeoutException: Connect to 10.191.43.74:8667 timed out
```

**根因:**

`knowledge/server/application-dev.yml` 中配置了 `python.domain: http://10.191.43.74:8667`，该地址为太保内网 IP，外网环境不可达。Python 向量服务实际运行在本机 `127.0.0.1:8667`。

**解决方法:**

```yaml
guyu:
  python:
    domain: http://127.0.0.1:8667
```

无需重新编译，只需 patch jar 包内的 yml 文件:

```bash
# 重新打包时注意 compress 类型
# 嵌套 JAR 内容（classes/）用 STORED (compress=0)
# yml/json 等配置文件用 DEFLATED (compress=8)
```

### 12.3 im-platform 实体字段映射错误（MyBatis-Plus 下划线）

**问题现象:**

```
GET /im-platform/robot/config/page
返回 500: {"code":500,"message":"系统繁忙，请稍后再试"}
```

**根因:**

`application-dev.yml` 中 `mybatis-plus` 配置了 `map-underscore-to-camel-case: false`，禁用了 MyBatis-Plus 的驼峰→下划线自动映射。但 `RobotConfig.java` 实体的 `tenantCode`、`robotType` 等字段没有显式 `@TableField` 注解，MyBatis-Plus 直接将字段名作为列名使用，生成 SQL `WHERE tenantCode = null` → `Unknown column 'tenantCode' in 'where clause'`。

**解决方法:**

在实体类中加 `@TableField` 注解显式映射：

```java
// RobotConfig.java
@TableField("tenant_code")
private String tenantCode;

@TableField("robot_type")
private Integer robotType;

// RobotApplyRelation.java（如有同样问题）
@TableField("tenant_code")
private String tenantCode;
```

修改后重新编译:

```bash
cd /data/guyu/source/guyu/im
mvn clean package -DskipTests
```

### 12.5 移除内网 UM 查询，改本地重复校验

**问题现象:**

前端员工录入页面 → UM 输入框每次输入都调 `/user/findByUserUm` 查内网 UM 系统 → 返回"无当前用户" → 无法录入新员工。

**根因:**

`UserServiceImpl.findByUserUm` 原来调用内网 `UmClientApi.queryUserByUm()`，在外网环境无法访问；且逻辑是"查不到就返回 null"，前端认为 UM 无效。

**解决方法（后端）:**

改为只查本地 `im_user` 表，UM 不存在 → 返回空 UserVO（不重复，可录入）；存在 → 返回用户信息（重复）:

```java
// UserServiceImpl.java findByUserUm()
@Override
public UserVO findByUserUm(String userUm) {
    // 只查本地 im_user 表，校验 UM 是否重复
    // 存在 → 返回用户信息（重复）
    // 不存在 → 返回空的 UserVO（不重复，可录入）
    User user = this.lambdaQuery().eq(User::getUserName, userUm).one();
    UserVO userVO = new UserVO();
    userVO.setUserName(userUm);
    if (user != null) {
        userVO.setNickName(user.getNickName());
        userVO.setEmail(user.getEmail());
        userVO.setDeptId(user.getDeptId());
        userVO.setDeptName(user.getDeptName());
    }
    return userVO;
}
```

**解决方法（前端）:**

`el-select` 远程查询组件改为普通文本框，移除 `remote-method` 调用，姓名/邮箱改为手动输入:

| `src/views/Console/components/staffDialog.vue` | 同上，姓名/邮箱从 `disabled` 改为可编辑，移除 `userNameChange` 自动填充逻辑 |
| `src/views/Staff/UserManage.vue` | 同 `ServiceManage.vue` |

### 12.7 KBImpl / RobotTeam 调用 `sysUserMapper.selectOne(LambdaQueryWrapper)` 报 NoSuchMethodError（2026-07-19 修复）

**与 §12.1 的区别**：
- §12.1 修的是 `AuthUtils.addKnowledgeTeamMember`（guyu-biz-stubs 桩类签名不一致）
- 本节修的是 `SysUserMapper.selectOne(LambdaQueryWrapper)`（接口继承 + mybatis-plus 泛型擦除问题）
- 两者症状都是 NoSuchMethodError，但根因完全不同，**不要混用解决方案**

**问题现象**：

浏览器访问 `http://101.33.212.119/csp-ai-webadmin/#/knowledge/home` 登录后，点"知识库"菜单，前端调用 `POST /admin-server/knowledge-bank/knowledgeBank/query` 返回：

```json
{"code": 500, "message": "系统繁忙，请稍后再试", "data": null}
```

`server.log` 中出现：

```
java.lang.NoSuchMethodError: com.paic.robot.workbench.mapper.SysUserMapper
    .selectOne(Lcom/baomidou/mybatisplus/core/conditions/query/LambdaQueryWrapper;)Ljava/lang/Object;
  at com.paic.csp.ai.knowledge.bank.service.impl.KnowledgeBankServiceImpl
      .getUser(KnowledgeBankServiceImpl.java:2601)
  at com.paic.csp.ai.knowledge.bank.service.impl.KnowledgeBankServiceImpl$$EnhancerBySpringCGLIB$$xxxx
      .getUser(<generated>)
```

**完整的 4 个调用点**（4 个 ServiceImpl 都有同样的 bug）：

| 文件 | 行号 | 上下文 |
|---|---|---|
| `knowledge-bank/.../KnowledgeBankServiceImpl.java` | 2601 | `public User getUser()` |
| `knowledge-bank/.../KnowledgeBankServiceImpl.java` | 2572 | `workBenchKnowledge()` |
| `robot-workbench/.../KnowledgeTeamServiceImpl.java` | 170 | `getUserByName()` |
| `robot-workbench/.../RobotTeamServiceImpl.java` | 436 | `getUserByName()` |

**根因（深度分析）**：

`SysUserMapper` 是空接口，仅继承 `BaseMapper<User>`：

```java
public interface SysUserMapper extends BaseMapper<User> {
}
```

`BaseMapper`（mybatis-plus 3.x 任意版本）的 `selectOne` 方法签名都是：

```
descriptor: (Lcom/baomidou/mybatisplus/core/conditions/Wrapper;)Ljava/lang/Object;
```

即**只接受 `Wrapper` 参数、返回 `Object`**。

但调用方代码（KBImpl 第 2601 行）写的是：

```java
LambdaQueryWrapper<User> queryWrapper = Wrappers.lambdaQuery();
...
User user = sysUserMapper.selectOne(queryWrapper);
```

由于 `LambdaQueryWrapper` 是 `Wrapper` 的子类，**javac 在编译时推断 `selectOne` 调用时**，生成的不是 `invokeinterface selectOne(Wrapper)Object`，而是 `invokeinterface selectOne(LambdaQueryWrapper)Object`——**保留了 LambdaQueryWrapper 作为参数类型**。

这是 javac 在**泛型方法在接口继承时的特定行为**，与 mybatis-plus 版本无关（3.3.2 / 3.5.3.2 都有这个问题）。

**为什么 7-13/7-14 当时能跑通**：

7-14 18:15 的 `knowledge-bank-1.0.0.jar` 是用 **mybatis-plus-boot-starter 3.3.2** 编译的，编译时类路径里同时存在 3.3.2 + 3.5.3.2 的 BaseMapper（maven 依赖冲突导致两个版本都在 classpath），3.3.2 时代 BaseMapper 有内部桥接方法处理 LambdaQueryWrapper 调用，所以 `KBImpl.selectOne(LambdaQueryWrapper)` 在运行时能解析到 3.3.2 的实现上。

7-19 `mvn clean package` 后，3.3.2 starter 被彻底清除，**只剩 3.5.3.2 的 BaseMapper**（无 LambdaQueryWrapper 重载），NoSuchMethodError 暴露。

**解决方法（Tier A：源码重编译，已验证）**：

涉及 5 个文件，全部为源码级修改，下次 `mvn clean package` 自动继承：

**1) 升级 mybatis-plus 统一版本**（避免 3.3.2/3.5.3.2 混存）：

```xml
<!-- /data/guyu/source/guyu/knowledge/pom.xml -->
<mybatisplus-spring-boot-starter.version>3.5.3.2</mybatisplus-spring-boot-starter.version>
```

**2) 在 `SysUserMapper` 接口加 default 桥接方法**（关键，返回类型必须为 `Object` 与字节码描述符严格匹配）：

```java
// /data/guyu/source/guyu/knowledge/robot-workbench/src/main/java/
//     com/paic/robot/workbench/mapper/SysUserMapper.java
package com.paic.robot.workbench.mapper;

import com.baomidou.mybatisplus.core.conditions.Wrapper;
import com.baomidou.mybatisplus.core.conditions.query.LambdaQueryWrapper;
import com.baomidou.mybatisplus.core.mapper.BaseMapper;
import com.paic.robot.workbench.entity.User;

public interface SysUserMapper extends BaseMapper<User> {
    /**
     * 桥接方法：把 sysUserMapper.selectOne(LambdaQueryWrapper) 委托给 BaseMapper.selectOne(Wrapper)
     * 原因：javac 在编译调用方时保留 LambdaQueryWrapper 签名作为参数类型，
     *      但 BaseMapper 只声明了 selectOne(Wrapper)，
     *      运行时 NoSuchMethodError: selectOne(LambdaQueryWrapper)Object
     * 注意：返回类型必须是 Object (raw type)，与 KBImpl 字节码调用的描述符一致。
     *      使用 <T> T 泛型会导致 javac 推断为 Object，运行时仍报 NoSuchMethodError。
     * 修复日期：2026-07-19
     */
    @SuppressWarnings("unchecked")
    default Object selectOne(LambdaQueryWrapper<User> queryWrapper) {
        return BaseMapper.super.selectOne((Wrapper<User>) queryWrapper);
    }
}
```

**3) 4 处调用方加显式 cast**（default 方法返回 Object，需在调用方做强转）：

| 文件 | 改动 |
|---|---|
| `KnowledgeBankServiceImpl.java:2601` | `User user = (User) sysUserMapper.selectOne(queryWrapper);` |
| `KnowledgeBankServiceImpl.java:2572` | `User user = (User) sysUserMapper.selectOne(queryWrapper);` |
| `KnowledgeTeamServiceImpl.java:170` | `User user = (User) sysUserMapper.selectOne(queryWrapper);` |
| `RobotTeamServiceImpl.java:436` | `User user = (User) sysUserMapper.selectOne(queryWrapper);` |

**4) 重新打包并重启**：

```bash
cd /data/guyu/source/guyu/knowledge
mvn clean install -DskipTests   # 编译所有子模块，重打 server.jar
# 重启 knowledge-server（端口 8089）
```

**5) 验证**：

```bash
# 接口必须返回 code:200
curl -sS -X POST "http://127.0.0.1:8089/admin-server/knowledge-bank/knowledgeBank/query" \
  -H "UserName: TSETPT913" -H "UserId: 42" -H "tenantCode: pacloud" -H "source: backend" \
  -H "Content-Type: application/json" -d '{"pageNum":1,"pageSize":10}'

# 期望响应: {"code":200,"message":"成功","data":{...}}
# 不能再有 NoSuchMethodError / IllegalAccessError
```

> **不要做的方案**（已踩坑）：
> - ❌ 在 jar 内 patch `SysUserMapper.class` 添加桥接方法 → 报 `IllegalAccessError`（独立 class 与原 interface 同存导致冲突）
> - ❌ 仅升级 pom 版本不修 SysUserMapper 接口 → javac 仍生成 LambdaQueryWrapper 签名，NoSuchMethodError 仍在
> - ❌ 仅修改 4 处调用方不加 cast → 编译报 "incompatible types: Object cannot be converted to User"
>
> **三层缺一不可**：① pom 升级 3.5.3.2 + ② SysUserMapper 加 default 桥接 + ③ 调用方加 `(User)` cast。

### 12.8 MySQL root 密码被改导致知识库所有 SQL 报 Access denied（2026-07-19 复盘）

**与 §12.7 的区别**：
- §12.7 是字节码问题（NoSuchMethodError），表现为"系统繁忙，请稍后再试"
- 本节是 MySQL 连接问题（Access denied），表现为 SQL 无法执行
- 两个 bug **叠在一起**导致 knowledgeBank/query 失败，必须都修了才好

**问题现象**：

`server.log` 中持续出现（短时间内可达数十万次）：

```
ERROR ... Access denied for user 'root'@'localhost' (using password: YES)
ERROR ... org.springframework.jdbc.CannotGetJdbcConnectionException: Failed to obtain JDBC Connection
```

通过 `mysqlbinlog` 查 binlog 定位密码修改时间：

```bash
mysqlbinlog --no-defaults --start-datetime="2026-07-16 15:30:00" --stop-datetime="2026-07-16 16:00:00" /var/lib/mysql/binlog.000005
# 找到: ALTER USER 'root'@'localhost' IDENTIFIED WITH 'mysql_native_password' AS '*6BB4837EB74329105EE4568DDA7DC67ED2CA2AD9'
# *6BB4837EB74329105EE4568DDA7DC67ED2CA2AD9 = sha1(sha1("123456")) → 密码被改成 123456
```

也可通过 MySQL 自带元数据查：

```sql
SELECT user, host, authentication_string, password_last_changed
FROM mysql.user WHERE user='root';
-- password_last_changed 应为修改时刻
```

**根因**：

MySQL root 密码在 7-15/7-16 期间从 `root` 被改成了 `123456`，但 `guyu/knowledge/server/src/main/resources/application-dev.yml` 里写的是 `password: root`，导致所有知识库 / 消息 / 机器人 / 工单相关 SQL 全部 Access denied。

**为什么知识首页登录还能进**：

`/admin-server/management-console/user/login` 走的是 **im-platform 的 user_login 接口**（8888 端口），不是 knowledge-server 直接查 MySQL，所以首页登录可以通。但登录后前端调 `knowledgeBank/query` 等接口经过 nginx → 8089（knowledge-server）→ MySQL，触发 Access denied。

**解决方法**（最小改动：保持共享资源不变，新建独立用户给新项目）：

**原则**：MySQL root 是这台机器所有服务共享的全局凭据，**绝不可随意修改**。正确做法：

**1) 恢复 MySQL root 密码到原值 `root`**：

```sql
ALTER USER 'root'@'localhost' IDENTIFIED WITH mysql_native_password BY 'root';
FLUSH PRIVILEGES;
-- 验证: mysql -uroot -proot -e "SELECT 1"  → 返回 1
```

**2) 改回 `application-dev.yml` 的 `password: root`**：

```yaml
# /data/guyu/source/guyu/knowledge/server/src/main/resources/application-dev.yml
spring:
  datasource:
    password: root   # 恢复原始值
```

**3) 重新打 jar 并重启**：

```bash
cd /data/guyu/source/guyu/knowledge/server
mvn package -DskipTests   # 注意：不要 mvn clean，避免清掉 backup
# 重启 knowledge-server
```

**4) 验证**：

```bash
curl -sS -X POST "http://127.0.0.1:8089/admin-server/robot-workbench/team/getTeamSpace" \
  -H "UserName: TSETPT913" -H "UserId: 42" -H "tenantCode: pacloud" -H "source: backend" \
  -H "Content-Type: application/json" -d '{}'
# 期望: HTTP 200 + {"code":200, ...}
# server.log 不应再有 Access denied
```

> **绝不要做的事**：
> - ❌ 改 MySQL root 密码来"适配"新项目的连接配置（影响所有依赖 MySQL 的服务）
> - ❌ 改 `/etc/` 下系统级配置去绕开密码问题
>
> **正确做法**：
> - 新项目需要不同数据库连接配置时，**新建独立 MySQL 用户**：`CREATE USER '<proj>'@'localhost' IDENTIFIED BY '...'; GRANT ON <proj>.* TO '<proj>'@'localhost';`
> - 然后改新项目自己的连接配置（application.yml、.env、config.py 等）使用新用户

**为新项目新建独立用户的范例**（SDD-project 场景）：

```sql
-- 在 MySQL root 下执行
CREATE USER 'sdd'@'localhost' IDENTIFIED WITH mysql_native_password BY '123456';
GRANT ALL PRIVILEGES ON `sdd`.* TO 'sdd'@'localhost';
FLUSH PRIVILEGES;

-- 验证隔离
mysql -usdd -p123456 sdd -e "SHOW TABLES;"         -- 应可见 sdd 库表
mysql -usdd -p123456 appdata -e "SHOW TABLES;"     -- 应被拒: Access denied
```

```bash
# /root/workspace/SDD-project/start_sdd.sh 启动时加环境变量
AI_NATIVE_PORT=8010 \
  AI_NATIVE_DB_USER=sdd \
  AI_NATIVE_DB_PASSWORD=123456 \
  nohup $VENV/bin/python flask_app.py > $LOGDIR/backend.log 2>&1 &
```

**这样修改后**：
- guyu 用 root/root，不受 SDD 部署影响
- SDD 用 sdd/123456，权限严格限定 sdd 库，不能访问 guyu 的 appdata
- 任何后续项目同理：先建独立用户，再配项目连接
