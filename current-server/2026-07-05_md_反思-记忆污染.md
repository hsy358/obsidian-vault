## 🔍 根因分析

### 直接原因
我把**"之前的服务器"（已废弃 101.35.52.96）的硬件配置**当成新服务器配置写进了文档。

### 深层原因

1. **vault 里有"老服务器"历史归档**：
   ```
   /root/vault/1-Projects/德勤/AI-Native/AgentSpace-部署/
   2026-07-05 - AgentSpace 部署踩坑与修复全集（归档版）.md
   
   第 6 节写：
   | 磁盘 | 59 GiB / 12 GiB 可用 | 建议 ≥20 GiB 可用 |
   | 内存 | 3.6 GiB / 2.2 GiB 可用 | 建议 ≥4 GiB |
   ```
   **这是历史正确的**——老服务器（轻量腾讯云）确实就是 3.6 GiB / 2 核。

2. **我没现场核查新服务器硬件**：写文档时直接凭印象"小内存腾讯云"，没跑 `nproc` / `free -h`。

3. **MEMORY.md 没有"新服务器硬件规格"章节**：之前讨论过老 IP（101.35.52.96 → 101.33.212.119），但**没明确记录硬件差异**。两台是不同的腾讯云 VM，**硬件完全不同**。

4. **GPT 训练偏差**：小内存腾讯云（2C3.6G）是"常见配置"，我下意识套用了。

## 🛠 改进措施（**永久生效**）

### 规则 1：写文档前**现场核查 6 项**

```bash
# 任何写"服务器 / 硬件 / 部署" 文档前必须跑：
nproc                  # CPU 核数
free -h                # 内存大小
df -h /                # 磁盘
hostname               # 主机名
cat /etc/os-release    # OS 版本
curl ifconfig.me       # 公网 IP
```

**输出贴进文档"硬件规格"章节**，不凭印象。

### 规则 2：MEMORY.md 必带"硬件规格"章节

- ✅ MEMORY.md 现在加了 **"🖥️ 服务器硬件规格"** 章节
- ✅ 包含新服务器 + 老服务器对照表
- ✅ 包含必查命令清单
- ✅ 标注"实测时间"（2026-07-05 14:37）

### 规则 3：碰到**"之前的服务器"**的引用时

- ❌ **不**复用老服务器的硬件 / 路径 / DB 配置
- ✅ 在文档里**明确标注**：*"本配置适用于新服务器 101.33.212.119（VM-0-13-ubuntu，8C30G）。如需老服务器参考，见 X 归档文档。"*
- ✅ 老服务器配置仅作为**历史归档**，**不再用于新部署决策**

### 规则 4：**配置文件 .env 实际值必查**

之前的"BETTER_AUTH_SECRET=paperc…cret"是简写。**实际值**就是 `paperclip-dev-secret`（默认）。
- ✅ 写文档时用 **`.env` 实际值**（cat /root/.paperclip/instances/default/.env）
- ❌ 不再用 `…cret` 简写或记忆里的版本

### 规则 5：反思笔记 **永久保存**

每次发生记忆污染 / 错误判断，**立即写反思笔记**到 vault `current-server/反思-*.md`，并：
- 写进 MEMORY.md 的"反思教训"章节
- 写进改进规则
- 永久保留（不归档到 4-Archives）
