## 关键决策记录

### 决策 1：用 GitHub API 下载，不用 git clone

- **场景**：腾讯云主机 GitHub 443 被屏蔽，`git clone` 卡死 134 秒
- **选择**：`curl api.github.com/repos/xerrors/Yuxi/tarball` （API 可达）
- **验证**：下载 5.7MB tarball，解压正常

### 决策 2：串行 build

- **场景**：首次 `--build` 8 小时后失败，buildx 日志报 `image already exists`
- **根因**：api + worker 共用 `yuxi-api:0.7.1.beta1` image 名，buildx 并发 build 时命名冲突
- **修法**：分四次 `docker compose build <service>` 串行执行，worker 复用 api image（0.2s skip）

### 决策 3：改 Yuxi 端口，不改其他服务

- **冲突**：`postgres:5432` 和 `redis:6379` 已被 RAGFlow / Langfuse 占用
- **方案**：`5433:5432` + `6381:6379`（主机端口换，容器端口不变）

### 决策 4：完整模式部署（不含 mineru/paddlex）

- **场景**：mineru-vllm 和 paddlex 需 NVIDIA GPU
- **方案**：保持默认 `docker compose up -d`（不带 `--profile all`），这两个 service 不启
- **影响**：聊天、基本 Agent 工作可用；**文档上传/PDF 解析不可用**（除非改外部 MinerU API）

### 决策 5：用 MiniMax 直连 API，不用 SiliconFlow

- **场景**：Yuxi 默认模型 `siliconflow-cn:Pro/MiniMaxAI/MiniMax-M2.5` 需 SiliconFlow key（未注册）
- **方案**：
  1. PUT `minimax-cn` provider + `is_enabled=true` + MiniMax key
  2. POST `/api/system/config` 改 `default_model=minimax-cn:MiniMax-Text-01`
  3. 同步改 `fast_model` + `content_guard_llm_model`
- **残留**：`embed_model` 和 `reranker` 仍用 siliconflow-cn（聊天不需要，但**知识库/RAG 功能需要**配 SiliconFlow key）

## 跟德勤项目的关联

**Yuxi 是德勤 AI Native MVP 的开源参考实现**，对应关系：

| Yuxi 能力 | 德勤 MVP 组件 |
|---|---|
| LangGraph 多智能体编排 | **Hermes Agent v0.14 dispatcher**（已选型） |
| Skills / MCP / SubAgents | 德勤 MVP 核心 |
| 沙盒（sandbox-provisioner） | Agent 工具执行环境 |
| 知识库 / RAG / Milvus / Neo4j | 组织知识管理 |
| 多租户 | 多业务线并行 |
| Langfuse 可观测集成 | Hermes trace |

**借鉴价值**：
1. **可插拔执行器抽象层** — Yuxi 把 agent / tool / skill / mcp 都拆成独立模块，Hermes 可以借鉴这种抽象
2. **前后端分离 + Vite dev mode** — Vue 3 + Pinia + Vite 启动 600ms，迭代快
3. **provider 抽象** — OpenAI 兼容协议下，20+ provider 灵活切换

**不借鉴**：
- 数据库表结构（Yuxi 偏 Postgres-first，Hermes 偏 Checkpoint-SQLite）
- Web UI（何大人用 Obsidian + Hermes dispatcher，不需要 web 控制台）
