不是段子，不是标题党。

这是上个月 OpenClaw 社区里流传的一个 Polymarket 量化交易案例——据说有人用龙虾搭了一套自动化交易 Agent，丢进去 50 美元做种子资金，48 小时后账户余额变成了 2950 美元。帖子下面炸了，光截图就转了上千次。当事人自己发了条帖子："睡了一觉，钱翻了。"

你可能会想：这跟我有什么关系？我又不是程序员。

**说实话，你不需要是。**

今天这篇文章，手把手带你用 Openclaw 从零搭建一套基础的炒股 Agent 团队。**不写代码，只写中文。** 你只需要会打字、会复制粘贴，就能搞定。搭完之后，每天早上 9 点半开盘，你的 AI 团队自动盯盘、自动分析、自动筛股、自动出报告，推送到你的飞书里。

话不多说，直接开始。

## 在动手之前，先搞懂一个核心概念

如果你之前看过我写的那些 OpenClaw 文章，这部分可以跳过。如果你是第一次接触——别走，两分钟就够。

Skills 是什么？

打个比方。你刚入职一家公司，领导不可能跟你说一句"去干活"就完了吧？至少得给你一份岗位培训手册：你的岗位职责是什么、工作流程是什么、遇到问题找谁、产出物交给谁、什么事情不归你管。

**Skill 就是你写给 AI 的岗位培训手册。没有它，你的龙虾就是个陪聊的。**

你跟没有 Skill 的龙虾说"帮我分析一下最近的 A 股行情"，它给你编一段看似专业的废话。为什么？因为它没有明确的工作流程，不知道从哪拉数据，不知道用什么指标，不知道输出格式该长什么样。它只能靠"猜"——而猜出来的东西，你敢用来做投资决策吗？**绝对不能。**

有了 Skill 就不一样了。你给它写清楚：去哪个 API 拉数据、用哪些技术指标分析、输出格式是什么、异常情况怎么处理。它就照着这套流程一步一步执行。**从"随机应付"变成"标准化作业"。**

这玩意在技术上长什么样？其实就是一个文件夹，里面放一个叫 `SKILL.md` 的 Markdown 文件。你用中文写清楚这个 AI 该干什么、怎么干、输出什么，它就按你的要求来。

不需要编程。会写 Markdown 就够了——**确实，连 Markdown 不会写也没关系，后面我会教你让 AI 帮你写。**

好，概念搞定，进入正题。

## 搭建炒股 Agent 团队：四步走

很多人一上来就想直接写 SKILL.md。

大错特错。

**大部分人的第一个 Skill，是一个什么都干、什么都不精的万能废物。**

为什么？因为你脑子里只有一个模糊的念头——"帮我炒股"——然后就一股脑塞进一个文件里。结果 AI 看完之后也是一脸懵：你要我做什么？盯盘？选股？写报告？还是帮你下单？全都干？那我先干哪个？数据从哪来？分析完了给谁？

一个什么都干的人，往往什么都干不好。AI 也一样。

正确的路径是这样的：

**模糊需求 → 需求拆解 → 逐个搭建并验证 → 串成流水线**

先想清楚，再一个一个搭，搭一个测一个，全部跑通了再串起来。

跟开餐厅一个道理。你不会先把厨师、服务员、收银员、采购全招进来，然后说"开始营业"。你会先让厨师试菜，菜过关了，再让服务员试跑流程，流程顺了，再接收银。每个环节单独验收，最后才开门迎客。

下面四步，一步一步带你走完。

## 第一步：把"帮我炒股"变成一份清晰的需求文档

还记得前面说的"万能废物"吗？大部分人打开 OpenClaw，第一句话就是"帮我搞一个炒股的 Skill"——然后 AI 真的给你搞了一个什么都干、什么都不精的东西。接口报错、逻辑瞎编、格式乱飞。因为从一开始它的工作范围就没有被定义过。

怎么避免？先写需求文档。

先别急着写 SKILL.md。先用一个 Prompt，**让 AI 帮你把模糊的需求拆解成清晰的需求文档。**

下面这个 Prompt 模板，你直接复制就能用。把最后的场景描述换成你自己的需求就行：

你是一个 AI Agent 架构师。我想在 OpenClaw 里搭建一个[场景描述]的 Skill 体系。

请帮我完成以下任务：

1. 把我的需求拆解为 3-5 个独立的子任务（每个子任务对应一个 Agent 角色）
2. 为每个子任务明确：
   - 角色名称和职责定义
   - 输入数据来源（API/文件/用户输入）
   - 输出格式和交付物
   - 需要使用的工具或 API
   - 触发条件（什么时候这个 Agent 该开始工作）
3. 设计这些 Agent 之间的协作流程（谁先谁后、谁给谁传数据）
4. 列出需要的外部数据源和 API
5. 给出安全注意事项

我的场景描述：[在这里填入你的具体需求]

示例填写：
我的场景描述：我想搭建一个炒股 Agent 团队，能够每天自动分析 A 股市场行情，筛选出 5 支潜力股，并给出买入/持有/卖出建议，最后把分析报告推送到我的飞书里。

![](https://mmbiz.qpic.cn/mmbiz_jpg/J1Cdba5GUc0cQIl8uBtICNMPvbwRMzeY8rrrV7icm2rrRcWZ8icznbEjqkicf3l1ob9huuIZQ3mRtYru3icAsSnM8yAgZDiczC1xkteJzKrFbxwc/640?wx_fmt=jpeg&from=appmsg)

把这个 Prompt 丢进 OpenClaw，它会回你一份详细到令人发指的需求拆解文档。以炒股场景为例，它大概会给你拆出 4 个 Agent 角色：

**1. 数据采集 Agent（盯盘员）**——负责每天从股票 API 拉行情数据，覆盖沪深 300 成分股，输出标准化 JSON。它的工作是"跑腿取货"，不做任何分析判断。

**2. 行情分析 Agent（分析师）**——负责对数据做技术面分析，算均线、MACD、成交量异动，识别形态，输出分析报告。它的工作是"看图说话"，只陈述事实不做推荐。

谁来最终拍板？**3. 策略决策 Agent（投资经理）**。综合分析师的报告和市场情绪数据，筛选 5 支潜力股，给出买入/持有/卖出建议，附上风险评级。这是整个团队的大脑。

最后一个——**4. 报告推送 Agent（秘书）**。别小看这个角色，它干的是"让老板看得懂"的翻译活。把投资经理的决策结果格式化成一份排版漂亮的报告，推送到飞书。

**四个岗位、四条职责边界、一套协作流程——从一句"帮我炒股"到一份完整的"招聘计划书"。**

而且这个 Prompt 是通用的。把"炒股"换成"竞品分析""客户舆情"——任何场景都能用：**先拆岗位，再写手册。**

接下来，不急着一口气招满四个人。先招第一个，让他上岗试试。

## 第二步：搭你的第一个 Agent —— 数据采集员

先从最简单的角色开始：数据采集员。它的工作就是跑腿取货——去 API 拉数据，存好，完事。不做分析，不做判断。

让 AI 帮你写操作手册

OpenClaw 有一个内置技能叫 skill-creator。你只需要跟它说"帮我创建一个数据采集 Skill"，它会根据你前面的需求拆解，自动帮你生成完整的 SKILL.md。就像你跟 HR 说"帮我把这个岗位的 JD 写出来"，HR 根据你给的职责描述，直接给你输出一份规范的岗位说明书。

在 OpenClaw 对话框里说：

"根据刚才的需求拆解，帮我创建第一个 Skill：股票数据采集。要求每天 9:30 开盘后自动获取沪深 300 成分股的行情数据，输出 JSON 格式。"

# 股票数据采集技能

## 触发条件
- 用户要求获取股票数据或市场行情
- 每日定时任务（9:30 开盘后触发）
- 被策略决策 Agent 调用时

## 工作流程

### 1. 获取行情数据
- 调用 scripts/fetch_stock_data.sh 获取当日 A 股行情
- 数据包含：开盘价、收盘价、最高价、最低价、成交量、换手率
- 默认获取沪深 300 成分股

### 2. 数据清洗与格式化
- 过滤停牌股票
- 计算涨跌幅、5 日/20 日均线
- 输出为 JSON 格式存入 workspace/data/daily/

### 3. 异常处理
- 如果 API 返回错误，等待 30 秒后重试，最多 3 次
- 如果数据缺失超过 10%，标记为异常并通知用户

### 4. 输出
- 文件：workspace/data/daily/YYYY-MM-DD.json
- 摘要：在对话中输出"今日数据采集完成，共 XX 支股票，XX 支涨 / XX 支跌"

这里有一个细节很重要——**description 字段。**

这个字段不是给你看的，是给 OpenClaw 看的。龙虾靠这段描述来判断"什么时候该调用这个 Skill"。你写得越清晰，触发就越精准。写得模糊，它就乱调，甚至不调。

你可以这样理解：description 就是这个员工的岗位职责说明。写得好，他知道什么活该接、什么活不该接。写得烂，他要么什么活都抢，要么什么活都不干。

注意看 description 里那句"不适用于：加密货币数据、美股数据、基金净值查询"。这不是废话。这叫**负面边界定义**——明确告诉 AI 什么不该干，比告诉它什么该干还重要。

**写得好的 description 是门卫，写得烂的 description 是摆设。**

文件写好了，放在 `~/.openclaw/skills/stock-data-fetcher/` 目录下：

~/.openclaw/skills/stock-data-fetcher/
├── SKILL.md
└── scripts/
    └── fetch_stock_data.py

scripts 里的辅助脚本你也可以让 AI 帮你写。跟 AI 说一句"帮我写一个从东方财富 API 拉取沪深 300 数据的 Python 脚本"，它五分钟就给你生成好了。

恭喜你——你刚才完成的这套流程："写 SKILL.md → 创建 Agent → 对话测试 → 验证输出"，就是搭建每一个 Agent 的标准动作。后面三个人，照着来就行。

**你的第一个 AI 员工，不是招来的，是你亲手教出来的。**

顺便说一个进阶小技巧。你可以给每个 Agent 配一个 SOUL.md，定义它的性格和工作风格——相当于给 AI 装了个良心。你是谁、你的原则是什么、你的底线在哪，全写进去。

比如分析师 Agent 的 SOUL.md 可以这么写：

# 你是一名 A 股技术分析师

## 身份
-专注 A 股市场技术面分析
-分析风格偏保守，注重风险控制
-不做基本面判断，只看技术指标

## 输出风格
-数据必须精确到小数点后两位
-每个结论必须附带数据支撑
-不确定的判断用"待观察"标注
-输出格式统一使用 Markdown 表格

## 禁忌
-不给出具体买入价位
-不预测涨跌幅
-不推荐个股，只做技术面陈述
- 不使用"必涨""稳赚"等误导性词汇

有了 SOUL.md，这个 Agent 就不只是一个执行工具——它有了自己的"职业操守"。不该说的不说，不该做的不做。你不用每次都叮嘱它"别给我推荐个股"，因为 SOUL.md 里已经写死了。

## 第三步：用同样的方法，搭剩下三个人

你已经掌握了"搭→测"的节奏。接下来三个 Agent 的创建步骤完全一样：让 skill-creator 生成 SKILL.md → 创建 Agent → 对话测试。

唯一的区别在两个地方：**模型选择**和**数据怎么接。**

行情分析师：第一次接力

分析师的 SKILL.md 结构跟数据采集员一样，但有一个关键区别：**它的输入不是 API，而是上一步存进去的 JSON 文件。** 你让 skill-creator 帮你生成时，要特别强调这一点——"读取 skills/stock-data-fetcher/data 目录下的最新 JSON 文件作为输入"。

在 OpenClaw 对话框里说：

"根据刚才的需求拆解，帮我创建第二个 Skill：股票行情分析。要求读取 skills/stock-data-fetcher/data 目录下的最新 JSON 文件作为输入。"

投资经理和秘书：照葫芦画瓢

剩下两个人的搭建流程完全一样，我不再展开了——唯一的区别是模型选择。

投资经理是整个团队的大脑，综合分析师报告做最终决策，用 Claude Sonnet（脑力活配好脑子）。秘书负责把决策结果排版成飞书日报然后推送，用 DeepSeek（体力活不用杀鸡用牛刀）。

**一个人什么都干，叫加班。四个人各干各的，叫体系。**

四个 Agent，各就各位。

🌊

四个 Agent 各自都能独立干活了。但你可能注意到——刚才的测试都是你手动触发的。你一个一个跟它们说"该你了"，它们才动。

那能不能让它们自动串起来？你不用管，它们自己接力跑？

能。这就是最后一步。

## 第四步：让四个人串成流水线

你得告诉它们谁先干谁后干、干完了通知谁。就像导演画分镜——第一场谁出镜、第二场谁接戏、台词说完了谁喊 Action，全在剧本里写好。

在 OpenClaw 对话框里直接说：

你：帮我配置一下这四个 Agent 的协作流程。数据采集完成后自动触发行情分析，分析完成后自动触发策略决策，决策完成后自动生成日报推送到飞书。

## 说几句掏心窝子的话

说实话，到这里我必须泼一盆冷水。

**AI 炒股赚钱这事，风险极大。**

前面提到的 Polymarket 案例，就算是真的，也是概率事件。量化交易的本质是赌概率，赢了的人到处发帖，亏了的人默默删号。**你看到的都是幸存者。**

四个安全要点你必须刻进脑子里：

**第一，绝对不要让 AI 自动下单。** 最终的买卖决策必须你自己来。把交易账户的 API Key 交给 AI，跟把银行卡密码告诉陌生人没有区别。这是底线。

**第二，AI 的分析结论只是参考，不是真理。** 它看的是历史数据和技术指标，它挡不住一条突发新闻把整个板块砸到跌停。

**第三，数据安全要把住。** 特别是那些免费的股票数据 API。**天下没有免费的午餐，你的数据可能就是它的午餐。**

**第四，先用模拟盘跑至少一个月。** 急什么呢？市场明天还在。

**你给它清晰的策略，它帮你 7x24 执行。你给它模糊的指令，它帮你精准地亏钱。**

说了这么多风险，不是要吓退你。恰恰相反——**知道边界在哪的人，才走得最远。**

## 你搭的不只是一个炒股工具

到这里，回头看看你做了什么。

你没有写一行代码。你做的事情是——想清楚需求、拆解岗位、写操作手册、建团队、设流程、定排班。

**这不是编程。这是管理。**

**你管的不是代码，是团队。只不过团队成员是 AI。**

而且这套逻辑，可以复制到任何场景。

想搭一个竞品监控系统？拆成信息抓取、变化检测、报告汇总三个 Agent，流程几乎不变。每天自动扫 10 个竞品官网，有新动态了自动总结、自动推送，你再也不用每天花半小时刷竞品。

**想象一下这个场景。**

早上闹钟 9:30 响了。你还在地铁上，耳机里放着播客。手机震了一下——飞书通知。

你点开一看，一份排版工整的日报已经安安静静地躺在群里了。标题、市场概览、五支推荐股、每支附带技术指标和风险评级，最后还有一段大盘情绪总结。

**你的四个 AI 员工，比你先到岗 15 分钟。**

你对象问你"又在看手机？"你说"我在开晨会。"

以前你是 AI 的用户，你问它答。

现在你是 **AI 团队的管理者**，你定流程，它们执行。

**从用户到管理者——这个身份切换，比你搭出来的任何一个 Skill 都重要。**
