## 02为什么 Karpathy 的 4 条不够用了？

Karpathy 最初总结的 4 条规则（Think Before Coding / Simplicity First / Surgical Changes / Goal-Driven Execution）非常精准。它们解决的是 AI **编码阶段**最基础、最高频的失败模式。

问题是，现在的 AI 编码早就不只是"帮我改一个函数"了。

它是跨多个文件、多个步骤、多个上下文执行的 Agent 工作流：读代码 → 改代码 → 写测试 → 跑验证 → 修复失败 → 继续重构。这个过程中，会出现 4 条规则完全覆盖不到的新问题：

![](https://mmbiz.qpic.cn/sz_mmbiz_png/yYic69VxxPmoibaicibN2vSeKODR9yQAP83AVnDxLDMeAdX5T6r5quibMr1eqxQ44o6gFn99mG6cIcPVb2HLnOe8uLAY5rNx3PicRBhQ1RfUdIpes/640?from=appmsg)
Agent工作流漂移**问题 1：上下文漂移。** 当一个 session 越来越长，模型会逐渐忘记之前试过什么，开始重复错误路径。你在第 30 轮对话里看到它重新试了一个你第 5 轮就否决的方案——这就是 token 预算失控的后果。→ 需要规则 6。

**问题 2：模型越界判断。** AI 开始用自己来判断"HTTP 503 要不要重试"这种确定性逻辑，而不是写代码来处理。它在替代你的代码。→ 需要规则 5。

**问题 3：冲突模式被"平均"。** 代码库里存在两套冲突的错误处理模式时，AI 很容易写出一套"两边都沾一点"的平均代码——既不像 A 也不像 B，但两边的缺点都继承了。→ 需要规则 7。

**问题 4：测试只是"看起来通过"。** AI 说"测试通过"时，你还要确认：它是不是跳过了某些测试？是不是只验证了表面返回值，而不是业务意图？→ 需要规则 9、12。

**问题 5：多步骤执行中的状态丢失。** 一个 5 步的重构任务，AI 在第 3 步就开始基于错误假设执行第 4 步。因为没有检查点机制，错误一路扩散到最后。→ 需要规则 10。

过去我用 AI 写代码时，差不多 **85% 的轮次**都会撞上这里面的某一种问题。轻一点的是多改一个空行，重一点的就是整条重构路线直接跑偏。

加了这套规则之后，这个比例降到了**不到 8%**。不是 AI 变聪明了，是它终于知道边界在哪了。

## 0312 条规则中英对照详解

下面逐条过一遍。每条我标了英文名、中文名、核心含义，以及我自己踩坑之后最有感触的一句解读。

我按"编码阶段"和"Agent 执行阶段"分了两组——前半部分是你写代码时需要的，后半部分是你让 AI 自己跑任务时需要的。

### 编码阶段规则（Rules 1-4）

这 4 条源于 Karpathy 的原始 CLAUDE.md，是地基。

### Rule 1: Think Before Coding（编码前先思考）

> 写代码之前，先说明你的假设、看到的歧义和不确定性。如果有多种解释，列出它们，而不是悄悄选一个。

**💡 一句话解读**：90% 的 AI 代码返工都不是因为代码写错了，而是因为它在错误的前提下开始写的——这一条强迫它在动手之前暴露理解。

**翻车现场**：你说"把这个接口优化一下"，AI 默默假设你指的是"改成异步"，然后一路写下去。实际上你只是想把返回字段缩减两个。沉默假设 = 时间黑洞。

### Rule 2: Simplicity First（简洁优先）

> 用能解决问题的最少代码。不写投机性功能，不为一次性需求建抽象层。

**💡 一句话解读**：AI 天生有"架构冲动"——它看不得一个 if-else 就能解决的问题，非要套三层工厂模式；这条规则是给它的过度工程欲踩刹车。

**翻车现场**：你说"加个开关控制新老逻辑"，AI 给你建了一个 Feature Flag 框架，带配置中心对接、灰度策略、回滚机制。你只是想要一个 `if (config.enableNew)`。

### Rule 3: Surgical Changes（外科手术式修改）

> 只改必须改的地方。不顺手"清理"相邻代码、注释或格式。每一行改动都要能直接追溯到用户请求。

**💡 一句话解读**：这是防止"正交性破坏"的硬约束——AI 特别喜欢在你让它改 A 文件时，顺手把 B 文件的格式也"修正"了，而它根本不知道 B 文件的格式是故意的。

**翻车现场**：你让 AI 修一个函数里的 bug。它修好了。但它还顺手把同一个文件里另外三个函数的缩进从 tab 改成了 space——然后你整个团队的 CI 炸了。

### Rule 4: Goal-Driven Execution（目标驱动执行）

> 把模糊任务变成可验证目标。定义成功标准，然后循环验证直到达成。

**💡 一句话解读**："帮我优化性能"不是目标，"让这个 API 的 P99 延迟从 300ms 降到 100ms 以下，并且所有现有测试继续通过"才是——这一条是防"我做好了但你没法验证"的死循环。

**翻车现场**：AI 说"优化完成"。你问"优化了什么？" AI 说"改了几个 N+1 查询"。你问"效果呢？" AI 说"代码看起来更好了"。这种对话在没规则 4 之前，我经历了不下 50 次。

### Agent 执行阶段规则（Rules 5-12）

这 8 条是面向现代 AI Agent 工作流扩展的，解决的是"能稳定完成一段复杂任务"的问题。

### Rule 5: Use the Model Only for Judgment Calls（只把模型用于需要判断力的地方）

这条的核心就一句话：**如果代码能回答，就让代码回答。** 模型只用来做分类、起草、摘要、从非结构化文本里提取信息——这些需要"判断力"的事。

我原本也觉得这条太理论，直到亲眼见到 AI 在一个 API 返回 503 之后，没有写任何重试代码，而是自己在那判断"嗯看起来是临时故障，我再调一次试试"。然后它重试了 11 次。

我看着 token 账单发呆。

这是最隐蔽也最危险的错误——AI 在用大模型的概率输出替代你的确定性代码。你把决策权外包给了一个不知道自己不知道什么的系统。

### Rule 6: Token Budgets Are Not Advisory（Token 预算不是建议）

> 单任务预算 4,000 token。单 session 预算 30,000 token。接近上限时，做总结和重启，而不是静默超支。

**💡 一句话解读**：长 session 是 AI 编码质量的隐形杀手——超过一定 token 后模型开始"遗忘早期决策"，然后重复试错；这条规则强迫你在质量崩盘之前重启上下文。

**翻车现场**：一个 session 开了 3 小时，聊了 200 轮。AI 在第 180 轮提出一个方案，跟你第 20 轮否决过的方案一模一样。你愤怒地问它"我不是说过这个不行吗？"——它已经不记得了。上下文窗口是满的，但有效信息早就被挤出去了。

### Rule 7: Surface Conflicts, Do Not Average Them（暴露冲突，不要折中平均）

> 当两个已有的模式冲突时，不要混合它们。选一个（优先选更新、测试更充分的），标记另一个留待后续统一。

**💡 一句话解读**：AI 面对冲突模式时的本能是"各取一半"——结果既不是 A 也不是 B，而是一个两边缺点都继承了的新变种；这条规则强迫它做选择而不是和稀泥。

**翻车现场**：你的项目里一半 API 用 `Result<T, E>` 返回，另一半用 `throw`。AI 写新接口时，搞了个 `Result<T, E>` 但里面还偷偷 `throw`——你 code review 的时候血压直接拉满。

### Rule 8: Read Before You Write（写之前先读）

> 写代码前，先读懂当前文件的 exports、直接调用方、以及明显相关的共享工具。如果现有架构不清晰，不要继续。

AI 最爱犯的错：假设代码库里已经有什么——然后自己另起炉灶再造一个。

你让它加一个格式化日期的功能。它写了一个全新的 `formatDate` 函数，带了 3 个参数和 6 种格式选项。实际上 `shared/utils` 里早就有了。

它没去读。它觉得"加个日期格式化嘛，能有多复杂"。这条规则就是给它加一个硬性步骤：**动手之前，先把上下文读了。**

### Rule 9: Tests Verify Intent, Not Just Behavior（测试要验证意图，而不只是行为）

> 测试必须表达行为为什么重要，而不仅是它做了什么。不要把浅层测试通过当成正确性的证明。

**💡 一句话解读**：AI 写测试的天赋是"测它能测的东西"而不是"测应该测的东西"——比如它会给 getter/setter 写单元测试，但不会测"当用户余额为负时提现应该失败"；这一条把测试从代码覆盖拉回到业务覆盖。

**翻车现场**：AI 说"测试全部通过"。你一看，3 个测试分别验证了：函数返回值不为 undefined、返回类型是 string、调用没有 throw。业务逻辑有没有 bug？不知道。但测试覆盖率 100%。

### Rule 10: Checkpoint After Every Significant Step（每个重要步骤后设置检查点）

这一类错误在长任务里特别常见。你让 AI 做三步重构：拆分大函数 → 提取公共逻辑 → 更新调用方。它做到第 2 步时，突然觉得"这种公共逻辑应该用装饰器模式"，然后就开始了一场完全不在计划内的重写。

等你发现的时候，已经跑了 50 轮了。

规则很简单：**每个重要步骤后，总结已完成、已验证、待处理。** 如果上下文丢了，停下来重述当前状态。这本质上是一个"强制同步"机制——你在每一步结束后把 AI 拉回来，确认它没有漂移。

### Rule 11: Match Codebase Conventions（匹配代码库约定，即使你不同意）

> 遵守现有约定优先于个人品味。命名、结构、测试、错误处理模式都要匹配。如果遇到有害约定，明确指出而不是静默分叉。

**💡 一句话解读**：AI 有自己的"代码品味"——它可能认为你的项目应该用 kebab-case 而不是 camelCase，然后默默在新文件里改风格；这一条告诉它：一致性 > 正确性。

**翻车现场**：你的团队用 `snake_case` 命名数据库字段，用 `camelCase` 命名 JS 变量。AI 觉得"不一致不好"，开始在 JS 文件里也写 `snake_case`。你和同事在代码评审时互相对视，确认不是对方改的。

### Rule 12: Fail Loud（大声失败）

> 如果有任何内容被静默跳过、不确定、或失败了，就不能说"已完成"。默认暴露不确定性，而不是隐藏它。

**💡 一句话解读**：这是 12 条里最容易被忽视也最重要的一条——AI 本能地倾向于说"做好了"而不是"这里我不确定"；这一条翻转了这个默认值，把"诚实暴露问题"变成硬性要求。

**翻车现场**：AI 说"迁移完成！"。你检查后发现：3 个文件被静默跳过了（"路径不存在"但它没说），2 个 import 路径改错了（"我没权限读那个文件"但它没提），1 个测试被注释掉了（"这个测试依赖外部服务"但它觉得不需要告诉你）。每一条单独看都不是大问题，合在一起就是你周末加班的理由。
