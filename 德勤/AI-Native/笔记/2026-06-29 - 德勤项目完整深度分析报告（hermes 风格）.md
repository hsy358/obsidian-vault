## 第 1 章 · 项目全景

### 一句话定位

> **AI Native 组织 Workspace**：面向管理咨询 × Physical AI 交付场景，把人、Agent、项目、目标、任务、上下文、执行器、审批、成本、交付物和组织记忆，统一收敛到同一个 Workspace 中运行。

### 与传统咨询工具的本质区别

| 维度 | 传统工具（Jira/Notion/ChatGPT） | AI Native 组织 Workspace |
|---|---|---|
| Agent 角色 | 外部工具，按需调用 | **数字员工**：有组织树、目标、预算、汇报关系 |
| 协作范围 | 人和人 | **人 + Agent + Agent** 三方协作 |
| 任务生成 | 手工拆解 | **目标驱动自动拆解**（Goal Tree → Task Tree） |
| 过程可见 | 只看日志 | **看板 + Trace + 证据链 + 审批** |
| 知识沉淀 | 文件归档 | **Skill / Blueprint / Memory** 沉淀为组织资产 |
| 安全治理 | 角色权限 | **沙箱 + 凭证脱敏 + 审计 + 客户隔离** |
| Physical AI | 不支持 | **Omniverse / Isaac Sim / cuOpt** 预留接口 |

**一句话总结**：我们做的不是"另一个 Jira + ChatGPT 集成"，而是**组织操作系统**——把咨询交付过程从"个人经验驱动"升级为"人机协同、过程可视、交付可控、能力可沉淀"的 AI Native 生产系统。

## 第 2 章 · 业务背景与痛点

### 管理咨询 × Physical AI 的 5 个核心痛点

| # | 痛点 | 现状表现 | 产品要解决的方向 |
|---|---|---|---|
| 1 | **知识孤岛** | 经验、模板、报告散落在个人电脑、群聊、网盘 | 项目级知识沉淀 + 交付物版本 + 可复用 Skill |
| 2 | **协作低效** | 策略/技术/数据/行业顾问靠会议+人工同步 | 多人+多 Agent 同 Workspace，任务/上下文/产物同步 |
| 3 | **过程不可见** | 只看到最终报告，看不到中间研究、判断、争议、证据 | Agent 对话流 + 工具调用 + 证据链 + 审查记录 |
| 4 | **重复劳动** | 资料整理、竞品分析、初稿撰写反复出现 | 研究/数据/写作/PPT/审查 Agent 自动化 |
| 5 | **数据安全压力** | 客户资料敏感，外部模型和工具调用存在泄露风险 | 项目隔离 + 沙箱 + 权限 + 审批 + 凭证脱敏 + 审计 |
| 6 | **规模瓶颈** | 高级顾问时间有限，依赖个人能力 | 把方法论编码为 Agent Skill，可复制交付能力 |

### 当前市场空白

| 现有方案 | 缺口 |
|---|---|
| **Jira / Asana / Trello** | 项目管理工具，不理解 Agent 协作、没有上下文工程、不沉淀咨询方法论 |
| **ChatGPT / Claude 网页** | 个人工具，无项目级 Workspace、无组织记忆、无治理 |
| **Dify / Coze / LangChain Studio** | Agent 编排工具，面向开发者，缺业务对象模型和看板 |
| **Vibe Kanban** | 偏 Coding 场景，不覆盖研究/数据/写作/PPT/审查 |
| **OpenAgents** | 偏消息协作，无目标树、无交付物管理、无审批治理 |

**核心洞察**：市场缺的不是"更好的 Agent 工具"，而是"**Agent 能进入真实项目过程**"的工作系统——让 Agent 真正成为团队成员，而不是临时工。

### 为什么"AI Native 组织 Workspace"是答案

> 未来的咨询交付不是"顾问用 AI 工具提效"，而是"**顾问 + 客户 + 项目经理 + 研究 Agent + 数据 Agent + 方案 Agent + PPT Agent + 审查 Agent + 执行器 + 知识系统**"在同一个 Workspace 协作完成。

**4 个关键升级**：

1. **从工具到组织**：Agent ≠ 工具调用，是有角色、职责、目标、预算的数字员工
2. **从任务到目标**：任务卡片追溯到客户目标 → 项目目标 → 阶段目标（Goal Tree）
3. **从检索到上下文工程**：每次执行都知道项目背景、历史结论、资料边界、输出要求
4. **从日志到治理**：过程可视、可控、可审计，避免黑箱自动化

### 必须自研的 9 项能力（NemoClaw 增强版 表 8 / 开发说明 表 16）

| # | 自研能力 | 为什么必须自研 | 一期做到什么程度 |

| 2 | **看板与工作台 UI** | 开源项目各有 UI，但不符合咨询交付过程和管理者视角 | 项目看板 + Agent 运行看板 + 交付物看板 3 类先跑通 |

| 5 | **咨询方法论模板** | 普通 Agent 平台不理解咨询项目阶段/交付物/审查规则/客户场景 | 沉淀 3-5 个咨询流程模板 + 5-10 个 Agent Skill |
| 6 | **Memory / Context 管理** | 组织记忆不能只靠 RAG；需明确项目记忆/组织记忆/个人记忆/权限边界 | 先做项目级 Memory：背景/目标/关键结论/交付物摘要 |
| 7 | **审计与安全控制平面** | NemoClaw 提供执行沙箱参考，但企业还需要权限/审批/日志/合规报告 | 保留操作日志/执行日志/审批记录；敏感操作需人工确认 |
| 8 | **Physical AI 任务模型** | 数字孪生/仿真/边缘 AI 任务不同于普通文档任务，需特殊状态和结果管理 | 一期仅做任务类型预留；后续接 Omniverse/Isaac/cuOpt |
| 9 | **执行器抽象层** | 避免 NemoClaw 锁死，确保可插拔架构 | Adapter 模式 + 心跳机制 + 统一日志格式 |

### NemoClaw 在本项目中的**精确角色**

> ⚠️ **NemoClaw 不是看板、不是主 Workspace、不是唯一技术依赖**

**它是**：
- ✅ **NVIDIA 官方执行层**（OpenClaw/Hermes/LangChain Deep Agents 的安全包装）
- ✅ **OpenShell 安全沙箱参考**（进程级隔离 + 网络策略 + 凭证保护）
- ✅ **Agent 生命周期管理**（onboarding + 推理路由 + Blueprint）
- ✅ **Physical AI 生态适配入口**（Omniverse/Isaac Sim/cuOpt 桥梁）

**它不是**：
- ❌ 主 Workspace / 主看板
- ❌ 多人协作工具
- ❌ 完整知识库 / Memory 系统
- ❌ 唯一依赖（必须有可插拔执行器架构，Claude Code/Codex/OpenClaw 同样可接）

**在 9 层架构中的位置**：第 4 层（安全沙箱）+ 第 8 层（Physical AI），不进入 1-3 层（产品控制平面 / 协作通讯 / 执行器网关）。

## 第 5 章 · 7 种看板（产品核心）

### 为什么看板是核心（不是附属 UI）

> 看板是 AI Native 组织 Workspace 的**核心控制界面**，传统 Kanban 只展示任务状态，本产品的看板还必须展示 **Agent 角色 + 执行器 + 上下文 + 工具调用 + 成本 + 审批 + 风险 + 证据 + 交付物**。

**3 个关键论点**：

1. **管理者视角**：咨询合伙人不关心"哪个模型"，只关心"项目现在推进到哪里了、有没有风险"
2. **过程可见**：传统工具只能看最终 PPT，看板要能看到"这份报告怎么从空白到交付"
3. **多角色融合**：一个看板同时服务项目负责人/项目经理/咨询顾问/审查人/管理员 5 类角色

### 7 种看板精确定义（NemoClaw 增强版 表 4 + 开发说明 表 11）

| # | 看板 | 回答什么问题 | 核心字段/功能 |
|---|---|---|---|
| 1 | **项目看板** | 这个项目现在推进到哪里了？ | 客户/项目/阶段/里程碑/任务状态/负责人/Agent 参与者/交付物 |
| 2 | **Agent 运行看板** | 哪些 Agent 在运行？它们在干什么？ | Agent 角色/所属项目/当前任务/执行器/心跳/状态/耗时/失败原因/暂停恢复终止 |
| 3 | **协作看板** | 人和 Agent、Agent 和 Agent 是怎么交互的？ | 频道/线程/@mention/委派关系/讨论记录/争议点/采纳结果 |
| 4 | **调用链看板** | Agent 调用了什么模型/工具/知识库？ | 模型/工具/检索来源/token/成本/延迟/错误/重试/调用输入输出摘要 |

### 一期看板实现的务实建议

### 数据模型与事件流设计要点

**核心对象关系链**：

```
组织/公司 → 客户 → 项目 → 阶段 → 目标 → 任务 → Agent 执行
→ 工具调用/消息/证据 → 交付物 → 审批 → 知识沉淀
```

**协作事件类型（AgentEvent Schema）**：

- `delegate`（委派）
- `challenge`（质疑）
- `tool_call`（工具调用）
- `artifact`（产物生成）
- `review`（审查）
- `approve`（批准）
- `reject`（退回）

**Memory 分层模型**（开发说明 §10）：

| 类型 | 内容 | 流向 |
|---|---|---|
| **Runtime RAM / 临时上下文** | 当前任务输入、工具结果、中间材料 | 仅服务当前执行 |
| **项目 Memory** | 项目背景、客户目标、关键结论、任务摘要、审批结果、交付物摘要 | 一期实现 |
| **知识沉淀候选** | 从项目 Memory 抽取的可复用内容（案例/方法/模板/Prompt/Skill/Blueprint）| 一期实现 |
| **正式组织知识** | 经专家/管理员审核，进入知识资产平台（Owner/版本/权限/证据链/本体标签）| 后续独立建设 |

## 第 8 章 · 风险与缓解

### 5 大风险矩阵

| # | 风险类型 | 风险描述 | 概率 | 影响 | 缓解方案 |
|---|---|---|---|---|---|
| 1 | **技术风险** | 过度依赖单一开源项目（特别是 NemoClaw 还是 Alpha） | 中 | 高 | 采用**可插拔执行器架构**；关键数据模型自研；第一阶段并行验证 Paperclip / AgentSpace / OpenAgents |
| 2 | **技术风险** | Agent 黑箱——只看到结果，看不到过程和责任链 | 高 | 高 | 设计 **AgentEvent Schema**（delegate/challenge/tool_call/artifact/review）；过程必须可见 |
| 3 | **范围风险** | 一开始同时做知识库/沙箱/数据治理/物理 AI，导致无法上线 | 高 | 高 | 严格限定 Phase 1 只跑通协作和执行闭环；后续阶段再补 |
| 4 | **人员风险** | 顾问不愿用——系统变成额外填报负担 | 中 | 高 | 围绕**交付物和实际提效**设计；少填表，多自动沉淀 |
| 5 | **合规风险** | 客户资料和凭证泄露；缺少审计 | 中 | 极高 | Phase 2/3 引入**沙箱 + 审批 + 凭证脱敏 + 网络控制 + 审计**；MVP 阶段做基本凭证不入 prompt 约束 |

### 失败回退路径

| 触发条件 | 回退路径 |
|---|---|
| **Agent 执行器适配困难**（Claude Code / NemoClaw 接口不稳定） | 退回最小 Adapter 模式，先用 CLI/HTTP Adapter 跑通，NemoClaw 推迟到 Phase 2 |
| **协作 UI 效果不佳**（用户嫌复杂） | 退回"项目任务看板 + Agent 列表 + 日志抽屉"的极简版，参考 Vibe Kanban |
| **第一阶段超期**（>2 个月） | 砍掉 Skill/MCP 完整 Registry；只保留基础 Agent 注册 + 任务看板 |
| **数据安全压力提前出现**（客户合规要求） | Phase 1 提前引入 Docker 沙箱 + 基础凭证脱敏；不必等 Phase 3 |
| **开源项目维护停滞**（如 Paperclip 停止更新） | 由于核心数据模型已自研，可独立运行；只需替换开源参考层的 Adapter |

## 第 9 章 · 汇报工具箱

### 跟老板沟通的标准话术（直接引用 NemoClaw 增强版 §13）

#### ❌ 不要说

> "我们准备用 NemoClaw 做看板。"

#### ✅ 应该说

> "我们未来做的是面向 **Physical AI 和管理咨询交付**的 **AI Native 组织 Workspace**。
>
> **看板、协作、任务、Agent 运行状态、调用链、审批和交付物沉淀** 是产品自身必须建设的核心能力。
>
> **NemoClaw 因为来自 NVIDIA 生态**，和 Physical AI、OpenShell 沙箱、GPU/仿真任务天然相关，所以适合作为**底层执行器与安全沙箱**的重要参考。
>
> 但我们**不会完全依赖 NemoClaw**，而会设计**可插拔执行器架构**，同时结合 **Paperclip、AgentSpace、OpenAgents、Langfuse、Mem0、Dify/RAGFlow** 等能力，逐步补齐组织协作、知识记忆、看板治理和 Physical AI 交付能力。"

### 跟客户讲解的 5 分钟版本（电梯演讲）

> **[0:00-0:30] 痛点开场**
> 当前咨询项目的痛点是：经验散落在个人电脑、过程不透明、交付物难以复用、Agent 协作缺乏治理。
>
> **[0:30-1:30] 产品是什么**
> 我们做的是 **AI Native 组织 Workspace**——一个把客户、项目经理、咨询顾问、研究 Agent、写作 Agent、审查 Agent 全部装在同一个工作空间里协作完成咨询交付的组织级平台。
>
> **[1:30-2:30] 核心差异化**
> 跟 Jira + ChatGPT 集成的区别是：**Agent 是有组织的数字员工**，有目标、有预算、有心跳、有汇报关系，不是一次性工具调用。
>
> **[2:30-3:30] 技术策略**
> 我们用开源组合 + 自研控制平面：Paperclip 管组织、AgentSpace 管 Workspace、NemoClaw 管 Physical AI 沙箱、Langfuse 管调用链——但核心数据模型和看板自研，避免被任何单一项目锁死。
>
> **[3:30-4:30] 落地节奏**
> 第一阶段 1-2 个月跑通端到端 Demo；第二阶段 2-4 个月看板产品化；第三阶段 4-8 个月加知识治理和安全；第四阶段 8-12 个月接 Physical AI。
>
> **[4:30-5:00] 价值**
> 我们的目标是让咨询交付**从"个人经验驱动"升级为"组织生产系统"**——每个项目沉淀为可复用 Skill，每个 Agent 像员工一样被管理。

### 关键决策点的"标准答案"

| 客户/老板可能问 | 标准答案 |
|---|---|
| **"为什么不用 ChatGPT Enterprise？"** | ChatGPT 是聊天工具，没有项目级 Workspace、没有 Agent 组织管理、没有看板治理、不能沉淀咨询方法论 |
| **"为什么不用 Coze / Dify？"** | 它们是 Agent 编排工具，面向开发者，没有业务对象模型（客户/项目/目标）和看板，不理解咨询交付过程 |
| **"为什么不用 Jira？"** | Jira 是任务管理工具，没有 Agent 协作、没有上下文工程、不支持咨询场景（研究/数据/写作/PPT/审查） |
| **"为什么不用 Notion AI？"** | Notion 是文档工具，AI 能力是外挂，没有 Agent 组织、目标驱动、过程可观测 |
| **"为什么做 6 个月这么久？"** | 因为我们要做的是**组织操作系统**，不是聊天机器人；MVP 1-2 个月先跑通协作闭环，后续分阶段补 |
| **"为什么不用 NemoClaw 做底座？"** | NemoClaw 是 NVIDIA 官方执行层（v0.0.58 Alpha），适合做 Physical AI 沙箱参考；不适合做 Workspace、看板、组织控制——我们用**可插拔架构**，NemoClaw 只是执行器之一 |
| **"怎么证明 ROI？"** | 衡量 5 维指标：效率（初稿时间）、质量（审查通过率）、复用（Skill 调用次数）、安全（审计完整率）、组织（并行项目数） |
| **"能不能加 Physical AI？"** | 可以，第 4 阶段（8-12 月）通过 Omniverse/Isaac/cuOpt 连接器接入，MVP 阶段任务模型已预留 |

### B. 关键术语表

| 术语 | 解释 |
|---|---|
| **AI Native 组织** | 把 AI Agent 纳入组织结构和业务流程，Agent 作为数字员工（有角色、职责、目标、预算）管理，而不是工具调用 |
| **Workspace** | 项目级协作空间，承载成员、Agent、任务、资料、消息、产物和上下文 |
| **Agent 组织树** | Agent 的角色、职责、汇报关系、Owner、预算和执行权限（来自 Paperclip 启发）|
| **Context Pack** | Agent 执行任务时携带的项目目标、资料、历史结论、约束和输出要求 |
| **AgentEvent** | 描述 Agent 行为的结构化事件（delegate/challenge/tool_call/artifact/review/approve/reject）|
| **Skill** | 可复用的 Agent 能力包（Prompt + 工具 + 步骤 + 模板 + 检查规则 + 输出格式）|
| **Blueprint** | 面向行业或项目类型的标准化交付模板（项目配置 + Agent 队伍 + 流程）|
| **Physical AI** | 数字孪生、机器人仿真、边缘 AI、GPU 推理、供应链优化等物理世界 AI 场景 |
| **OpenShell** | NVIDIA 的开源 Agent 沙箱运行时（NemoClaw 的底层）|
| **Harness** | Agent 执行框架的总称（含 Claude Code / Codex / OpenClaw / Hermes 等）|
| **MCP** | Model Context Protocol，模型上下文协议（用于 Agent 与工具/资源的标准连接）|
| **AgentRouter** | 多执行器统一路由（来自 AgentSpace 启发）|
| **BYOA** | Bring Your Own Agent，允许用户自带 Agent 接入平台（来自 Paperclip 启发）|
| **Heartbeat** | Agent 周期醒来检查任务、执行、汇报的机制（来自 Paperclip 启发）|
| **Goal Tree** | 客户目标 → 项目目标 → 阶段目标 → 任务卡片的层级结构 |
| **Project Memory** | 项目级运行时记忆：背景、客户目标、关键结论、任务摘要、审批结果、交付物摘要 |
