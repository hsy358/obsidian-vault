# NemoClaw 增强版速查（沟通用）

> **目的**：跟老板/客户/同事讲清楚"AI Native 组织 Workspace 是什么、NemoClaw 在哪、不在哪、为什么"
> **完整内容**：见同名 docx（54 段 + 12 表）｜**本速查只放沟通要点**

---

## 🎯 一句话核心判断

> **看板是本产品必须实现的核心能力；NemoClaw 是 NVIDIA 生态下的执行器、安全沙箱和生命周期管理参考，不是主 Workspace、不是看板产品，也不应成为唯一技术依赖。**

---

## 📌 5 个修订重点（v0.3 增强了什么）

1. **产品目标**：建设面向管理咨询 × Physical AI 交付的 AI Native 组织 Workspace
2. **看板目标**：7 种看板（项目/Agent/协作/调用链/审批风险/交付物/知识沉淀）
3. **NemoClaw 作用**：OpenShell 沙箱 + 生命周期 + 推理路由 + 安全策略 + NVIDIA 生态适配
4. **NemoClaw 边界**：**不是主工作台 / 不是多人协作看板 / 不是完整知识库 / 不应成为唯一依赖**
5. **落地路线**：先 Agent 协作 → 看板 → 知识/治理 → Physical AI 深度集成

---

## ❓ NemoClaw 6 问 6 答

| Q | A |
|---|---|
| NemoClaw 是什么？ | NVIDIA 开源 reference stack，让 always-on AI agents 安全运行在 OpenShell sandbox |
| 为什么重要？ | NVIDIA 生态，Physical AI 场景的沙箱/执行器参考 |
| 适合放在哪一层？ | **执行器 + 沙箱层**，不是主 Workspace 层 |
| 不能替代什么？ | **产品看板 / 多人协作 / 组织管理 / Memory / 知识治理 / 业务控制平面** |
| 我们的策略？ | 可插拔执行器架构，NemoClaw / Claude Code / Codex / OpenClaw / LangGraph 都能接入 |

## 🧩 7 种看板（产品必须自研）

| 看板 | 回答什么问题 |
|---|---|
| **项目看板** | 项目现在推进到哪里？ |
| **Agent 运行看板** | 哪些 Agent 在跑？在干什么？ |
| **协作看板** | 人和 Agent / Agent 和 Agent 怎么交互？ |
| **调用链看板** | Agent 调了什么模型/工具/知识库？ |
| **审批风险看板** | 哪些结果需要人工确认？哪些行为有风险？ |
| **交付物看板** | 最终产物和中间产物在哪里？ |
| **知识沉淀看板** | 哪些内容能复用为组织资产？ |

---

## 📊 NemoClaw 长处 / 弊端 / 补齐（7 维度速览）

| 维度 | 长处 | 弊端 | 我们补什么 |
|---|---|---|---|
| 生态匹配 | NVIDIA 生态贴近 | 偏 NVIDIA 锁死 | 执行器抽象层 |
| 安全执行 | OpenShell sandbox | 不等于企业级安全 | 自研安全控制平面 |
| Agent 生命周期 | onboarding + 维护 | 不等于可视化看板 | Agent 运行看板 |
| Blueprint | 声明式配置 | 不是咨询方法论 | 自研咨询 Blueprint |
| 推理路由 | 路由推理思路 | 不覆盖全局成本 | 统一模型网关 |
| Physical AI | Omniverse/Isaac 适配 | 不提供完整工作台 | Physical AI 项目看板 |
| 成熟度 | NVIDIA 官方背书 | 仍是 early preview/alpha | 先验证，不锁死 |

## 🛣️ 4 阶段路线（速览版）

| 阶段 | 时间 | 重点 |
|---|---|---|
| **1. MVP 跑通** | 1-2 月 | 多 Agent / 多执行器 / 任务看板 / 基础协作 + 交付物 |
| **2. 看板 + 协作产品化** | 2-4 月 | 7 种看板 + 协作消息 + 调用链/成本可观测 |
| **3. 知识/Memory/安全治理** | 4-8 月 | 项目/组织 Memory + 知识库 + 沙箱策略 |
| **4. Physical AI 深度集成** | 8-12 月 | Omniverse/Isaac/cuOpt + 仿真任务看板 |

**关键原则**：不要在第 1 阶段把知识治理、数据治理、复杂沙箱一次性做满。

---

## 🏗️ 9 层架构（自研 + 开源组合）

```
┌──────────────────────────────────────┐
│  1. 产品控制平面       ← 自研（核心） │ ← Paperclip / AgentSpace 参考
│  2. 协作通讯层          ← 自研       │ ← OpenAgents 参考
│  3. 执行器网关层        ← 自研       │ ← Paperclip heartbeat
│  4. 安全沙箱层          ← 开源       │ ← NemoClaw / OpenShell ⭐
│  5. 编排与流程层        ← 自研+开源  │ ← LangGraph / CrewAI
│  6. 可观测层            ← 开源       │ ← Langfuse / OpenTelemetry
│  7. 知识记忆层          ← 自研+开源  │ ← Mem0 / Dify / RAGFlow
│  8. Physical AI 层      ← 开源       │ ← Omniverse / Isaac / cuOpt
│  9. 业务交付场景                  │
└──────────────────────────────────────┘
```

**NemoClaw 定位**：**第 4 层**（安全沙箱）+ **第 8 层**（Physical AI 生态适配），不进入 1-3 层。

---

## 🗣️ 跟老板沟通的标准话术（直接用）

### ❌ 不要说
> "我们准备用 NemoClaw 做看板。"

### ✅ 应该说

> "我们未来做的是面向 Physical AI 和管理咨询交付的 **AI Native 组织 Workspace**。
>
> **看板、协作、任务、Agent 运行状态、调用链、审批和交付物沉淀** 是产品自身必须建设的核心能力。
>
> **NemoClaw 因为来自 NVIDIA 生态**，和 Physical AI、OpenShell 沙箱、GPU/仿真任务天然相关，所以适合作为**底层执行器与安全沙箱**的重要参考。
>
> 但我们**不会完全依赖 NemoClaw**，而会设计**可插拔执行器架构**，同时结合 **Paperclip、AgentSpace、OpenAgents、Langfuse、Mem0、Dify/RAGFlow** 等能力，逐步补齐组织协作、知识记忆、看板治理和 Physical AI 交付能力。"

## 📚 开源项目符合度（8 个速览）

| 项目 | 复用部分 | 符合度 | 主要缺口 |
|---|---|---|---|
| **Paperclip** | Agent 组织控制平面 | 高 | 不是咨询工作台 |
| **AgentSpace** | 人 + Agent Workspace | 高 | 需验证生产化 |
| **OpenAgents** | 频道/线程/DM | 中高 | 治理和知识弱 |
| **NemoClaw** | NVIDIA 执行层/沙箱 | 中高 | 不覆盖看板/Memory |
| **LangGraph** | 复杂 Agent 流程 | 中高 | 没产品 UI |
| **Langfuse** | 调用链 trace | 中高 | 不管任务/组织 |
| **Mem0 / Dify** | 记忆/知识/RAG | 中 | 需咨询行业化 |
| **Vibe Kanban** | Coding Agent 看板 | 中 | 偏研发场景 |

## 🛠️ 9 个必须自研的能力

1. **统一产品数据模型**（客户/项目/目标/任务/Agent/执行器/消息/产物/知识/审批/审计）
2. **看板与工作台 UI**（7 种看板）
3. **Agent 执行器网关**（可插拔，NemoClaw 只是之一）
4. **协作事件模型**（委派/质疑/采纳/审批/工具调用）
5. **咨询方法论模板**（3-5 个流程模板 + 5-10 个 Agent Skill）
6. **Memory / Context 管理**（项目级 Memory）
7. **审计与安全控制平面**（权限/审批/日志/合规）
8. **Physical AI 任务模型**（任务类型预留 + 后续接 Omniverse/Isaac）
9. **执行器抽象层**（避免 NemoClaw 锁死）

## 🎯 MVP 一期范围（1-2 个月）

| MVP 能力 | 具体 |
|---|---|
| ✅ 一个组织 Workspace | 成员/Agent/任务/交付物 |
| ✅ 一个项目任务看板 | 6 状态（待拆解→待执行→执行中→待确认→已完成→失败） |
| ✅ 一个 Agent 运行看板 | 角色/执行器/心跳/日志/暂停/终止 |
| ✅ 2+ 执行器接入 | Claude Code / Codex / OpenClaw / NemoClaw 至少 2 类 |
| ✅ 基础协作消息 | 人和人、人和 Agent 线程 |
| ✅ 基础调用日志 | 任务/执行器/工具/时间/失败 |
| ✅ 人工审批节点 | 关键交付物必须人工确认 |
| ✅ 咨询交付 Demo | 选一个咨询场景跑通端到端 |

**MVP 成功标准**：能完成一个咨询项目从需求 → 初稿 → 审查的端到端演示；5-10 个 Agent Skill 可用。
