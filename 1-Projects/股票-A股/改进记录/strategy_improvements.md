## 2026-06-26 整体复盘（AI 产业链 v3 推荐）

### 推荐 v3 总览（5 只）
- 推荐日：2026-06-24｜跟踪期：10 个交易日｜当前：3/10
- 6-25 收盘：平均 +4.58%（胜 4/5）
- 6-26 盘中：平均 -1.93%（胜 1/5）
- 关键现象：**一天内由全盈转全亏**

### 今日 5 只表现 vs 板块
| 标的 | 6-26 涨跌 | 所属板块 | 板块今日涨跌 |
|---|---|---|---|
| 立讯精密 | -9.80% | 消费电子/AI 终端 | 电子信息 -4.67% |
| 工业富联 | -8.79% | AI 服务器代工 | 电子信息 -4.67% |
| 通富微电 | -7.08% | 半导体封测 | 电子信息 -4.67% |
| 中国巨石 | +4.27% | 玻纤/光伏上游 | 玻璃 +1.97% ✅ |
| 中天科技 | -10.00% | 光通信/海缆 | 通信设备 |

### 根因分析
1. **行业集中度过高（最大问题）**：
   - 5 只里 4 只（立讯/工业富联/通富微电/中天）都跟电子/通信强相关
   - 当电子板块 -4.67% 时，5 只组合几乎全军覆没
   - 唯一正收益的中国巨石（玻纤）跟其他 4 只相关性低

2. **没有大盘风控（第二大问题）**：
   - 今天次新股 -6.03%、电子信息 -4.67%——明显是大盘风险事件
   - 推荐组合里没有"如果大盘跌 -1% 自动减仓"的规则
   - 完全是裸奔状态

3. **止损纪律执行了（做得对）**：
   - 立讯精密（-10.58%）触及预设止损 68.0 ✅
   - 工业富联（-8.56%）触及预设止损 72.0 ✅
   - **说明预设止损位合理**，问题不在止损，在板块集中

4. **基本面没错（短期）**：
   - AI 产业链长期逻辑（Rubin 平台、GB300、苹果新机）没变
   - 是市场风险偏好下降导致杀估值，不是逻辑证伪

### 改进措施（下次 AI/电子板块推荐必须执行）
1. **行业分散度硬约束**：单板块最多 2 只，5 只组合覆盖 ≥3 个板块
2. **大盘熔断规则**：
   - 上证指数当日跌 >1% → 组合减仓 30%
   - 上证指数当日跌 >2% → 组合减仓 50%
   - 跌破 MA20 → 全部清仓观望
3. **止损位动态调整**：
   - 大盘弱势时止损位上移 3%（避免洗出去）
   - 大盘强势时止损位下移 2%（给波动空间）
4. **跟中国巨石一样找"非相关性"标的**：
   - AI 产业链推荐时，强制配 1-2 只上游材料/海外业务占比高的对冲票

### 二、为什么 v3 失败（5 大根因，按 Yuxi Review 准则排序）

#### 根因 1：**没有 verifiable goal**（Yuxi 准则 4：Goal-Driven Execution）
- ❌ 推荐时只说"5 只 AI 产业链股"
- ❌ 没有"什么情况下算推荐失败"的失败标准
- ❌ 没有"自动触发止损"的可验证机制
- ✅ 改进：v4 必须把"verifiable goal"写出来（每只带 -8% 止损硬规则 + 每日 15:30 自动复盘）

#### 根因 2：**过度设计 + 不执行**（Yuxi 准则 2：Simplicity First）
- ❌ 6-26 写过 5 条改进措施（行业分散 + 大盘熔断 + 止损动态 + 配对冲票 + 跌破 MA20 清仓）
- ❌ 一条都没真执行（pure speculative feature）
- ✅ 改进：v4 只保留 3 条**能立刻验证**的硬约束（详见下文）

#### 根因 3：**跟踪流程 cron 中断**（系统性失败）
- ❌ status="completed" 但 daily_review 只写了 4/10
- ❌ 6-29 ~ 7-4 整周断档（端午 + MiniMax API 过载连环 crash）
- ✅ 改进：把 cron 从"agentTurn 调模型"改成"纯 Python 脚本 + 仅推送环节调模型"

#### 根因 4：**行业集中度过高**（组合方差问题）
- ❌ 5 只里 4 只电子/通信强相关
- ❌ 板块 -4.67% 时全军覆没（6-26 反思过但没改）
- ✅ 改进：v4 硬约束：5 只覆盖 ≥3 个板块

#### 根因 5：**止损位形同虚设**（执行问题）
- ❌ 6-26 写过"立讯 -10.58% 触发止损 68.0、工业富联 -8.56% 触发止损 72.0"
- ❌ 但实际是从 6-26 一直拿到 7-8，整整 9 个交易日没减仓
- ✅ 改进：止损位 = 基线 × 0.92（一刀切 -8%），写在 recommendations.json schema 里强制

### 三、v4 推荐策略设计（Simplicity First：只 3 条硬约束）

#### 硬约束 1：**板块分散度**
- 5 只覆盖 ≥3 个板块（相关性硬约束）
- 单板块最多 2 只
- 推荐理由里必须写"板块 + 相关性备注"

#### 硬约束 2：**每只预设 -8% 止损位**（自动写进 schema）
- 止损价 = baseline × 0.92
- 写入 `operation_plan.stop_loss` 字段
- 每日 15:30 复盘 cron 自动检测：触及止损 → 立刻写失败反思 + 通知何大人
- 不靠"我觉得"判断，靠 if-then 自动触发

#### 硬约束 3：**大盘熔断（轻量版）**
- 当日上证收盘跌幅 >1.5% → 整组标记 `risk_level: high`，**次日不加仓**
- 跌破 MA20 → 整组 `risk_level: critical`，**次日必须减仓 50%**
- 只 2 条大盘规则，比 v3 的 4 条更少更清晰

#### 推荐流程（每日 9:00 盘前）
```
1. 拉今日板块涨跌排名（电子 / 通信 / 新能源 / 医药 / 消费 / 金融）
   → verify: 拿到 ≥5 个板块数据
2. 在 ≥3 个不同板块里各挑 1-2 只
   → verify: 板块分散度 ≥3，单板块 ≤2
3. 每只给 baseline_price + 板块 + 相关性 + 止损位（基线 × 0.92）
   → verify: 5 只全部含 operation_plan.stop_loss
4. 写进 recommendations.json
   → verify: schema 校验通过
5. 推送给何大人（15:30 收盘后）
```

#### 跟踪流程（每日 15:30 收盘后）
```
1. 拉每只当日收盘价
2. 计算盈亏 %
3. if 任意一只触及止损（pnl ≤ -8%）：
   → 自动写 strategy_improvements.md 失败案例
   → 自动通知何大人"某只触及止损，建议减仓"
4. if 大盘跌 >1.5%：
   → 标记整组 risk_level
5. 跟踪满 10 个交易日 → 自动归档
```

### 四、cron 修复方案（避开 MiniMax API 过载）

**问题**：当前 14 个 cron 任务里 11 个 `error` 状态，全部是 `FailoverError: AI service overloaded`，连续失败 8-12 次。

**方案**：把 cron payload 从 `agentTurn`（agent 跑全流程）改成 `systemEvent`（仅触发信号），让 Python 脚本独立跑：

```bash
# 旧方案（被 API 过载击垮）
cron payload: agentTurn 完整 prompt → agent 跑全流程

# 新方案（确定性 + 鲁棒）
cron payload: systemEvent 短消息
  → Python 脚本：拉数据 → 写文件 → Git 同步
  → 仅推送环节调 agent（可选，失败不阻断主流程）
```

**优先级**：P1，明天做。
