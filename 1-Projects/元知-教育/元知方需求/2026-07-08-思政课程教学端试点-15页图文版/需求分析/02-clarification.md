## 1. 需求摘要

**一句话**：把"思政课教师 + AI 平台"协同的 8 步流程标准化：老师只在 2 个关键节点（结构确认 + 内容审核）拍板，其余由"元知团队 + 平台系统"批量完成。

## 2. 目标与非目标

### 2.1 业务目标

| 目标 | 衡量方式（v1 建议） |
|---|---|
| 完成 1-2 门新时代思政课程试点闭环 | 至少 1 门课通过 5 项验收标准 |
| 沉淀标准资料模板与交付模板 | 资料模板 1 套 + 交付模板 1 套（覆盖 9 类资产）|
| 验证元知教学端工作的落地可行性 | 试点复盘报告 + 推广建议 |

### 2.2 非目标（本次试点不做）

- ❌ 学生端（学情分析、互动教学、考试管理）
- ❌ 多门课程并行（试点仅 1-2 门）
- ❌ 跨校/跨区域推广（v1 不涉及）
- ❌ 校级数据大盘（管理空间部分）

## 4. 业务对象与生命周期

### 4.1 核心对象

| 对象 | 关键属性 | 状态机 |
|---|---|---|
| **课程** | 名称/目标/学时/对象/大纲 | 立项 → 资料齐备 → 图谱确认 → 资产审核 → 交付 |
| **资料包** | 资料清单 / 分类 / 版本 | v0.1（上传中）→ v1.0（已归档）→ v1.x（迭代）|
| **知识图谱** | 4 类子图 / 节点数 / 关系 | 构建中 → 已生成 → 老师确认 → 锁定 |
| **教学资产**（9 类）| 资产类型 / 模板 / 风格 / 版本 | 草稿 → 初稿 → 待审 → 已审 → 定稿 |
| **课程包** | 资产列表 / 版本 / 验收状态 | 打包中 → 验收中 → 通过 → 归档 |
| **审核记录** | 资产 ID / 审核人 / 意见 / 状态 | 待审 → 通过 / 退回 / 回调 |

### 4.2 课程对象完整状态流转

```
[立项申请]
  ↓ (PM 接收)
[资料提交中]
  ↓ (资料齐备)
[诊断中] → (资料不足 → 退回 [资料提交中])
  ↓ (诊断报告通过)
[图谱构建中]
  ↓ (图谱 v1.0 完成)
[老师第一次确认] → (修改 → 回到 [图谱构建中])
  ↓ (确认通过)
[资产生成中]
  ↓ (资产包 v1.0 完成)
[老师第二次确认] → (修改 → 回到 [资产生成中] | 回调 → [老师第一次确认])
  ↓ (确认通过)
[打包中]
  ↓
[验收中] → (验收不通过 → 回到 [资产生成中])
  ↓
[已交付] → (v2.0 触发 → 回到 [诊断中] 或 [图谱构建中])
```

### 4.3 关键状态迁移规则

- **v1.0 → v1.1**：老师第二次确认后局部修改 → 自动升级到 v1.1
- **v1.x → v2.0**：政策变化 / 学科发展 / 重大内容更新 → 需重新走完整 8 步
- **v1.x → 回调**：第二次确认中"问题需较多修改" → 回到第一次确认节点

## 6. 主流程

```
立项申请 → 资料提交 → 诊断 → 图谱 → 第一次确认 → 资产生成 → 第二次确认 → 打包 → 验收 → 交付
                                                                                       ↓
                                                                                  (v2.0 触发)
```

## 7. 分支流程

| 分支点 | 分支条件 | 处理 |
|---|---|---|
| 资料不足 | 必交材料缺项 | 标注缺失项，先用现有资料生成初稿 |
| 第一次确认未通过 | 修改意见 | AI 调整 → 再次提交确认 |
| 第一次确认需补充资料 | 资料不全 | 上传课程资料或说明补充口径 |
| 第二次确认局部修改 | 修改意见可控 | 按意见更新相关资源版本 |
| 第二次确认回调 | 问题需较多修改 | 回调到第一次确认节点 |
| 验收不通过 | 不符合 5 项标准 | 回到资产生成节点 |

## 8. 边界与异常场景

| 场景 | 处理 |
|---|---|
| 老师无历史课件 | 先用现有资料生成初稿，再补充修正 |
| 资料格式不规范（PDF 扫描件 / 图片）| 提示格式要求，建议重传 |
| 老师长时间未确认 | 触发提醒（24h / 48h / 72h 升级）|
| 老师意见相互冲突 | 标注后再进入下一轮（争议内容处理）|
| 资产生成失败（模型/工具异常）| 记录失败原因 + 重试 + 升级人工处理 |
| 课程包打包超时 | 单课超时 24h → 触发 PM 介入 |

## 9. 业务规则

- **BR-01**：老师必交材料 8 类缺一不可（最低要求层），缺项需明确标注
- **BR-02**：知识图谱未完成确认前，不得进入资产生成步骤
- **BR-03**：老师第一次确认未通过，不得进入资产生成
- **BR-04**：9 类资产推荐生成顺序为顺序依赖（前序未完成，后序不开始）
- **BR-05**：所有交付物必须有版本号（v1.0/v1.1/v2.0）
- **BR-06**：课程包交付必须通过 5 项验收标准
- **BR-07**：价值引领点缺失时，资产不得进入"已审"状态
- **BR-08**：争议内容必须标注，不得自动进入下一轮

## 10. 假设

- **A-01**：试点期间 1-2 门课的老师有时间参与 2 次确认（每门 1-2 天）
- **A-02**：元知方提供平台访问权限给老师/校方代表
- **A-03**：8 类必交材料中如缺历史课件，老师可先口头说明
- **A-04**：知识图谱构建依赖元知方已训练的 AI 引擎
- **A-05**：9 类资产生成能力由元知方提供（我方不需自建）
- **A-06**：试点阶段数据存储在元知云端，本地不持久化
