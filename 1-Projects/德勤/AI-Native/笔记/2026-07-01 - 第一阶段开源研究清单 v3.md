## 6. 平凯 Loop 🆕

- **官方地址**：https://loop.pingkai.cn
- **定位**：SaaS 形态的"Agent 协作工作区"（不是自托管产品，但 UX 设计已验证）
- **第一阶段研究目标**：
  - 看它如何用"**工作区 + 频道 + 话题 + 任务**"做协作最小单元
  - 看它如何做 **@ 范围权限管控**（普通消息不群发 / @all 显式广播 / 个人私有不外溢）
  - 看它如何做 **投递诊断快照**（哪些 Agent 命中/排除/唤醒/原因）
  - 看它如何用 **任务并行 + 多 Agent 协同执行**（Loop 的核心叙事：20x-50x 效率）
  - 看它执行器后端选型：**Codex / Claude Code / OpenCode**（跟德勤选型一致，是业界共识）
- **建议重点资源**：
  - 主站：https://loop.pingkai.cn（已完成）
  - Quickstart：https://loop.pingkai.cn/docs/quickstart（已完成）
  - 公众号文章：[[2026-07-01 - 什么是 Loop？什么是平凯 Loop？]]
  - 对照分析：[[2026-07-01 - 平凯 Loop vs 德勤 v0.3 对照架构图 (mermaid)]]
  - 主站功能列表：[[2026-07-01 - 平凯 Loop 主站功能列表]]
- **主要借鉴层**：
  - **R3**：执行器后端选型确认（业界共识：Codex / Claude Code / OpenCode 三选一）
  - **R3**：机器/智能体/运行实例 三层拆分 → 执行器抽象
  - **R4**：协作最小单元 = **任务板 + 话题**（这是 v0.3 R4 必须照搬的核心）
  - **R7**：@ 范围权限管控 + 投递诊断快照（v0.3 R7 设计参考）
- **当前机器判断**：
  - **试用**（5 分钟注册 + 跑通 5 步上手）
  - **只读文档**（不再额外下载源码，因为是 SaaS 闭源）
  - **不部署**（不可能部署 SaaS）
- **不建议完整借鉴的原因**（为什么不能直接用 Loop 代替 v0.3）：
  - **SaaS 形态** — 不符合 R9 模块化 + 单独部署原则
  - **无审批节点** — 咨询项目必须人确认关键交付物
  - **封闭数据模型** — 不能接入 v0.3 已有 8 张核心表
  - **可观测不可见** — Loop 内部有，外部用户看不到 Langfuse 级 trace
- **一句话结论**：
  - **平凯 Loop 是 R3 + R4 + R7 的 UX 真理（最小协作单元 + 投递控制），但 v0.3 必须自研后端，不能用 SaaS 替代。**
