# 第一阶段开源研究清单 v2

> 目标不是“替代 Hermes”，而是为德勤项目第一阶段的 **R1 / R3 / R4 / R7 / R9** 提供可借鉴的工程实现。
> 当前机器策略遵循：**能轻装就轻装，重系统只读代码，不为了单个研究对象破坏现有 Hermes / OpenClaw 链路。**

## 二、P0 主线

## 1. Paperclip

- **仓库**：`paperclipai/paperclip`
- **定位**：多 Agent 并存的组织级工作平台，重点不是某个单独 agent，而是 **agent adapter + 组织治理 + 统一数据模型**
- **第一阶段研究目标**：
  - 看它如何组织 `company / workspace / goal / budget / audit` 这一类上层模型
  - 看它如何设计多 adapter 并存接口
  - 看它如何处理治理边界，尤其是 `no-remote-git`、本地工作区持久化边界、run-level error
- **建议重点目录**：
  - `packages/adapters/`
  - `packages/adapter-utils/`
  - `packages/db/`
  - `packages/shared/`
  - `server/`
  - `AGENTS.md`
- **主要借鉴层**：
  - **R1**：统一数据模型 / 组织树 / 预算 / 审计
  - **R3**：adapter 抽象接口、执行上下文、结果契约
  - **R7**：治理边界、权限与远端 Git 限制
  - **R9**：模块拆分方式（server/ui/packages）
- **当前机器判断**：
  - **建议：只读代码，不部署**
  - 原因：第一阶段主要价值在结构和接口，不在于把整套产品跑起来
- **一句话结论**：
  - **Paperclip 是 R1 + R3 的主参考，也是 R7 / R9 的重要工程参考。**

## 2. LangGraph

- **仓库**：`langchain-ai/langgraph`
- **定位**：执行流编排框架，适合做状态机式 agent 执行后端
- **第一阶段研究目标**：
  - 看 `StateGraph` 如何表达 research / analyze / approval / deliver 这类阶段流
  - 看 checkpoint / memory / recover 机制
  - 把它当成“执行器内部引擎”，不是拿来替代 Hermes
- **建议重点目录 / 能力点**：
  - `StateGraph`
  - `MemorySaver`
  - conditional edges
  - checkpoint 恢复机制
- **主要借鉴层**：
  - **R3**：统一执行器接口背后的一个可选 runtime
- **当前机器判断**：
  - **可部署，且已轻量验证通过**
  - 已装包、已跑 demo、适合继续保留为参考实现
- **一句话结论**：
  - **LangGraph 不是主控制器，而是 R3 中最适合做状态化执行流后端的参考件。**

## 3. Hermes

- **仓库 / 系统角色**：Hermes Agent
- **定位**：当前方案里的 **主执行器 / 主控制入口 / 主编排器**
- **第一阶段研究目标**：
  - 梳理 `chat / gateway / sessions / skills / memory / hooks / kanban` 在德勤架构里的职责边界
  - 明确哪些能力直接复用，哪些需要在上层再包一层德勤域模型
- **建议重点能力**：
  - `chat`
  - `gateway`
  - `sessions`
  - `skills`
  - `memory`
  - `hooks`
  - `kanban`
- **主要借鉴层**：
  - **R3**：主执行器入口
  - **R4**：会话 / 协作 / 评论 / 任务同步
  - **R5**：Memory provider 抽象
- **当前机器判断**：
  - **已部署在用，继续作为主线研究对象**
- **一句话结论**：
  - **Hermes 不是候选项，而是已确定的主控层。**

## 4. OpenClaw

- **仓库 / 系统角色**：OpenClaw
- **定位**：第二执行器 + 外部 gateway 适配层 + 多入口实验对象
- **第一阶段研究目标**：
  - 看它如何承接外部消息入口与 agent 执行链路
  - 看它如何作为 Hermes 之外的平行执行器，被统一抽象到同一层
- **建议重点目录 / 能力点**：
  - gateway / channel / plugin 相关部分
  - `sessions_spawn` 这类可供执行器抽象借鉴的接口
- **主要借鉴层**：
  - **R3**：第二执行器 adapter
  - **R4**：多入口协作与消息路由
- **当前机器判断**：
  - **已部署在用，不需要额外重部署**
- **一句话结论**：
  - **OpenClaw 的价值不是替代 Hermes，而是验证“多执行器并存 + 多入口接入”这条路。**

## 5. AgentSpace

- **仓库**：`HKUDS/AgentSpace`
- **定位**：Human + Agents 共用 workspace 的协作平台，核心价值在 **AgentRouter / workspace / remote daemon / sandbox / deploy 分层**
- **第一阶段研究目标**：
  - 看 `AgentRouter` 如何统一接 Hermes / OpenClaw / Codex / Claude Code 一类 runtime
  - 看 workspace 协作模型如何承载 owner / role / shared workspace
  - 看 remote daemon 如何支持远程执行器拆分
  - 看 sandbox 如何做风险分级与安装计划
  - 看 deploy 目录如何拆 nginx / postgres / systemd / daemon
- **建议重点目录**：
  - `apps/web/`
  - `packages/daemon/`
  - `packages/db/`
  - `packages/domain/`
  - `packages/services/`
  - `packages/sandbox/`
  - `deploy/`
- **主要借鉴层**：
  - **R3**：AgentRouter / runtime routing / session propagation / event normalization
  - **R4**：workspace 协作模型
  - **R7**：sandbox / risk-aware operation / confirmHighRisk / audit
  - **R9**：daemon + web + db + deploy 的分层部署结构
- **当前机器判断**：
  - **只读代码，不建议完整部署**
- **不建议完整部署的原因**：
  - 当前机器 `Node` 版本是 22，而 AgentSpace 推荐 24
  - 明确依赖 PostgreSQL 16，而当前未装 `psql`
  - 机器资源偏紧：`2 vCPU / 3.6 GiB RAM / 剩余磁盘约 12G`
  - 现有机器上已经跑 Hermes / OpenClaw / nginx 等服务，再加 web + daemon + PG 风险偏高
- **当前正确策略**：
  - **研究级**：读 `README`、读 `packages/daemon`、读 `packages/services`、读 `packages/sandbox`、读 `deploy/`
  - **可选轻验证**：后续只 clone 代码，不起 PG，不全量安装，只看关键接口
  - **当前不做**：`npm run setup && npm run dev:web` 这种完整拉起
- **一句话结论**：
  - **AgentSpace 值得作为“结构与协作层参考”深读，但不适合现在在这台服务器完整部署。**
