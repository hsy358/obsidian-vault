# PDF 进知识库前，先过这一关
![](https://mmbiz.qpic.cn/mmbiz_jpg/b8T1AoFtlQQ5EO1Beia9gzYcCoOmsEXdg3X91BqcPZMwKElEJR5xHwhIYjJAsnrCnYMgdyykV3NlvMnS6nb2iatkZ9BykOgbs4CZRicMMhOyAg/640?from=appmsg)

很多 RAG 项目失败，不是模型不会答，而是资料进库时已经脏了。

一份扫描合同、一篇双栏论文、一张带表格的年报，丢进普通 OCR 后经常变成错序文本、丢公式、表格散架、页眉页脚混进正文。后面再换多强的模型，也是在拿坏食材做饭。

Allen AI 的 `olmOCR` 就是专门卡在这一关的工具。

![](https://mmbiz.qpic.cn/sz_mmbiz_png/b8T1AoFtlQTxCRm1d2rambJU1w8ZXia6NKTvh5TdmItLpQbWFsPdfpZsdS9oQwtiaqN7YOjpLwrHeTSswc0VnicbEKicT4OeqFJSwdFMcAXtbzU/640?from=appmsg)

## 它瞄准的不是识字，而是给 LLM 喂干净材料

olmOCR 的定位很清楚。它把 PDF、PNG、JPEG 这类文档转成干净、可读的 Markdown 或纯文本，重点支持公式、表格、手写内容和复杂排版。

README 里最打动我的一句不是功能清单，而是「natural reading order」。文档解析最麻烦的地方就在这里。人眼看双栏论文时知道先读左栏再读右栏，知道图注不是正文，知道页眉页脚应该扔掉；机器一旦读错顺序，RAG 检索出来的片段就会像被打乱的拼图。

olmOCR 用 7B 参数的视觉语言模型做这件事，所以它不是轻量脚本。官方也直接写了，本地跑需要 GPU，至少 12GB 显存，并建议准备 30GB 磁盘空间。换来的好处是，它不只想识别字符，还想把页面结构线性化成模型能吃的文本。

![](https://mmbiz.qpic.cn/sz_mmbiz_png/b8T1AoFtlQTHKFljM38j6zgvqm0SAHMkR4SWlbfsXiaE5Mv9aypNdtTN40YemNeuc215b4VGFLJzVBD4aKljk3nuE4JVb8CvVibXftBUoMfAs/640?from=appmsg)

## 真正有意思的是评测和训练都开出来了

很多 OCR 工具的问题不是不能用，而是你不知道它在自己的文档上到底错在哪里。olmOCR 把这一点做得比较工程化。

它随项目提供 `olmOCR-Bench`，覆盖 1400 份文档、7000 多个测试用例，维度包括 ArXiv、旧扫描数学、表格、页眉页脚、多栏、长小字等。v0.4.0 的 overall 分数是 82.4±1.1，和 README 里列出的 Chandra OCR、Infinity-Parser、PaddleOCR-VL、MinerU、Mistral OCR API 放在同一张表里比。

这个分数不是唯一结论，但表格本身很有价值。对做企业知识库的人来说，最怕的是只看几张漂亮 demo。合同、发票、论文、投标书、扫描档，每类文档的坑都不一样。评测维度拆开以后，你才知道某个工具是表格强，还是旧扫描强，还是只在干净论文上好看。

## 成本账比想象中友好，部署账不一定

olmOCR README 写了一个很直接的成本点。它的目标是每 100 万页转换成本低于 200 美元，约人民币 1430 元。这个数字对大规模资料入库很关键。很多团队不是没有 OCR 预算，而是没法接受每次重建索引都像烧钱。

但别把它理解成开箱即用的桌面工具。本地 GPU 路线要装 CUDA、vLLM、字体和 PDF 渲染依赖。Docker 镜像也有带模型版本，大约 30GB。它更适合有工程能力的团队，把文档解析做成一条稳定 pipeline，而不是临时拿来处理几份 PDF。

好在它也支持远程推理。你可以把 `olmOCR-2-7B-1025-FP8` 放到外部 vLLM 或兼容 OpenAI API 的服务上，再用命令行把 PDF 批量丢进去。官方还列了 Cirrascale、DeepInfra、Parasail 这类已测试 provider。也就是说，个人电脑不一定要扛下模型，团队可以把解析能力做成共享服务。

## 国内团队该把它当作一把尺子

如果你已经在用 MinerU、PaddleOCR-VL、Mistral OCR 或自研解析链路，olmOCR 未必马上替代它们。中文文档、版式规范、内网部署、合规要求，都会影响最后选择。

但它很适合拿来校准预处理层。把同一批合同、论文、飞书导出的 PDF、扫描件跑一遍，看 Markdown 顺序、表格保留、页眉页脚清理和失败率。很多 RAG 系统的问题，只有把解析结果摊开看，才会发现模型背了不该背的锅。

我会把 olmOCR 推荐给两类人。第一类是做企业知识库、论文库、法规库的团队，尤其是资料来源复杂、PDF 占比高的场景。第二类是正在评估文档 OCR 方案的工程团队，它的 benchmark、训练代码、pipeline 代码都能当参考。

不太适合的是只想偶尔识别几页 PDF 的用户。对他们来说，部署成本太高，直接用现成 OCR 服务更省心。
