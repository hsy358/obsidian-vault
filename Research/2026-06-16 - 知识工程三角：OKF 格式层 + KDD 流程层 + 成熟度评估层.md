## 1. TL;DR —— 一句话

**AI 原生企业的知识工程 = 三个支柱**：
- **格式层**（OKF 解决"知识怎么存"才能互通）
- **流程层**（KDD 解决"知识怎么用"才能进决策）
- **评估层**（5 级成熟度解决"现在到哪一步"才能定位短板）

任何想用 LLM 把组织内部知识用起来的团队，都得同时回答这三个问题。**三角三足**才能稳定。

## 2. 三篇文章的核心模型

### 2.1 OKF（Open Knowledge Format）—— 格式层

**发布方：** 谷歌 / `GoogleCloudPlatform/knowledge-catalog`（v0.1，2026-06）  
**解决的问题：** AI agent 需要的内部知识（数据表 / Wiki / 文档 / 注释 / 工程师脑子）散落各处、格式互不兼容、每次搭 agent 都重造轮子。

**OKF 的极简设计：**

```text
- 一个 bundle = 一个 Markdown 目录
- 每个文件 = 一个「概念」（数据表 / 指标 / API / 手册）
- 文件路径 = 概念的唯一标识
- 每个文件 = YAML frontmatter（结构化字段）+ Markdown 正文（自由内容）
- 强制字段：只有 type 一个
- 概念之间用普通 Markdown 链接互相引用
```

**三个设计原则：** 最小约束 / 生产者消费者解耦 / 格式 ≠ 平台

**三个参考实现：** 数据丰富 agent / 静态 HTML 可视化 / 3 个样例 bundle（GA4、Stack Overflow、比特币）

### 2.2 德勤 2026 KDD 金字塔 —— 流程层

**发布方：** 德勤 2026 趋势报告（战略流程智慧坊 公众号转述）  
**解决的问题：** 传统自动化（人定规则 + 流程机械执行）→ AI 原生流程（数据驱动 + 智能体自主决策 + 实时自优化）

**KDD 金字塔（自下而上）：**

| 层级 | 核心价值 | 落地关键 |
|---|---|---|
| **K** 知识层 | 沉淀业务规则、SOP | 流程数字化、经验标准化 |
| **D** 数据层 | 打通全链路数据 | 实时采集、治理统一 |
| **D** 决策层 | AI 自主判断、执行 | 智能体部署、闭环反馈 |

**关键判断：** 没有 K 层的规则沉淀 → AI 跑不出来；没有 D 层的数据打通 → AI 学不到东西；K+D 都有了 → 决策层才能跑通智能体。

**落地挑战：** 数据割裂 / 人才缺口 / ROI 周期长  
**三个标杆案例（数字需核实）：** 美的南沙（节拍 6 秒、库存降 30%）/ 科沃斯（人效 20 倍）/ 中信泰富（流程周期缩 30%）

## 3. 三角理论：格式层 + 流程层 + 评估层

### 3.1 三足互补

三篇文章并排放，能看出**三层互补关系**：

| 维度 | OKF（格式层） | 德勤 KDD（流程层） | 5 级成熟度（评估层） |
|---|---|---|---|
| 回答的问题 | 知识**怎么存**才能互通？ | 知识**怎么用**才能进决策？ | **现在到哪一步**了？ |
| 核心抽象 | 文件（Markdown + frontmatter） | 层级（K → D → D） | 5 个等级（L1-L5） |
| 强制项 | type 字段 | K 层规则 + D 层数据 | 达标线（覆盖率/线上化率） |
| 关注时间 | 静态（一次写好可读） | 动态（实时自优化） | 阶段性（逐级跨越） |
| 反对什么 | 平台锁定 / 专有 SDK | 季度人工优化 / 静态规则 | 跳级 / 走捷径 |
| 类比（建筑） | 砖头和水泥的**标准** | 怎么把砖头**砌成**会自我修复的房子 | 装修**工程监理**（检测质量） |

### 3.2 三角架构图

```text
              ┌──────────────────────────────┐
              │   评估层：5 级成熟度          │
              │   "现在到 L几了？             │
              │   短板在哪？下一步是 L几？"    │
              └──────────────────────────────┘
                       ↑ 定位           ↑ 衡量
                       │                │
   ┌───────────────────┐                ┌──────────────────────┐
   │ 格式层：OKF        │ ←─约束─→      │ 流程层：KDD          │
   │ "知识怎么存"        │                │ "知识怎么用"          │
   │ 砖头和水泥标准      │                │ 砖头砌成会修复的房子  │
   └───────────────────┘                └──────────────────────┘
```

### 3.3 三角的协同关系

| 场景 | 三角如何联动 |
|---|---|
| **启动期** | 评估层说「你这是 L1」（不达标）→ 格式层说「快用 OKF 规范文档」→ 流程层说「用 KDD 梳理关键流程」 |
| **升级期** | 评估层说「从 L2 → L3」→ 格式层说「打通跨 bundle 链接」→ 流程层说「线上化率要从 60% 升到 80%」 |
| **AI 化期** | 评估层说「到 L4 了」→ 格式层说「所有概念要有 type=metric」→ 流程层说「智能体部署到决策层」 |
| **成熟期** | 评估层说「到 L5 啦」→ 格式层说「bundle 越织越密」→ 流程层说「人工干预<20%，自优化闭环」 |

### 3.4 我的综合判断

**单有 OKF（好格式）** → 知识库好看但不会动；  
**单有 KDD（好流程）** → 智能体有决策逻辑但读不到一致的知识；  
**单有成熟度（好评估）** → 知道差距但不知道怎么改。  

**三角必须同时存在**：

```text
知识层（K）:  OKF 规范的 Markdown bundle    ←  格式层解决"读得到、读得懂"
        ↓
数据层（D）:  实时采集 + 治理统一           ←  跨 bundle 链接
        ↓
决策层（D）:  AI agent 闭环反馈             ←  流程层解决"用了能学"

（上面三层都看：5 级成熟度是「GPS」，告诉你「你到 L几、下一站是 L几」）
```

## 4. 和何大人现有工作流的关联

### 4.1 与 agentic-stack 记忆系统的映射

何大人目前的记忆架构（`/root/.agent/memory/`）已经无意中实现了 KDD 的雏形，但**没有评估层**：

| 5 级成熟度 | KDD 层级 | agentic-stack 对应 | OKF 字段（建议）|
|---|---|---|---|
| **L1 初始** | (无序) | 散落的 hardcoded rules | （无 type 字段） |
| **L2 显性** | K 知识层 | `semantic/LESSONS.md` | `type: lessons-catalog` |
| **L3 系统打通** | D 数据层 | `episodic/AGENT_LEARNINGS.jsonl` | `type: episodic-event` |
| **L4 AI 嵌入** | D 决策层 | `candidates/` + `graduate/reject` 工具 | `type: candidate` |
| **L5 AI 原生** | (闭环) | `tools/recall.py` + 自动 reflect | （需要闭环反馈） |

**当前位置评估：L3 → L4 之间**——K 和 D 层已有，D 决策层有人工 graduate，但**没有自优化闭环**。

**对照观察：** 何大人的 `lessons.jsonl` 每条已有 `conditions`、`rationale` 字段——这相当于 OKF 的 `tags` + `description` 字段的隐式版本。

### 4.2 用 5 级模型评估各资产成熟度

| 何大人资产 | 5 级评分 | 下一级路径 |
|---|---|---|
| `MEMORY.md`（人话速记）| L2 | L3：加 OKF frontmatter，顶部声明 `type: persistent-memory` |
| `memory/YYYY-MM-DD.md`（每日日志）| L2 | L3：加 `type: daily-log` + `tags: [topic1, topic2]` |
| `公众号文章/*.md` | L2 | L3：补 `type: wechat-article`（现有 frontmatter 缺 type 字段） |
| `Research/*.md` | L2 | L3：加 `type: research-report` + `topics: [...]` |
| `03-资源/PPT/*.pptx` | L1 | L2：加 metadata 文件（PPT 本身是二进制，OKF 不适用） |
| `lessons.jsonl` | L3 | L4：实现 AI 自动 graduate（基于 confidence × evidence） |
| `tools/recall.py` | L3 | L4：加 telemetry，记录「recall 后是否被用」作为反馈 |

### 4.3 与 MEMORY.md / vault 的映射

| 何大人文件 | 建议 OKF 化 |
|---|---|
| `MEMORY.md`（人话速记） | 顶部加 frontmatter：`type: persistent-memory`，`scope: cross-session` |
| `memory/YYYY-MM-DD.md`（每日日志） | 加 `type: daily-log`，`tags: [topic1, topic2]` |
| `公众号文章/2026-06-16 - XXX.md` | 已隐式有 `source: wechat`（可补 `type: wechat-article`） |
| `Research/*.md` | 加 `type: research-report`，`topics: [...]` |
| `03-资源/PPT/*.pptx` | （二进制，OKF 不适用，但路径本身可入库 OKF） |

### 4.4 工具视角：OKF + KDD + 5级成熟度怎么用到小助

**OKF 视角**（已有 70% 基础）：
- ✅ 已经是 Markdown + frontmatter（OKF 同款）
- ✅ 路径做概念标识（`/root/.agent/memory/semantic/LESSONS.md` 是 unique 路径）
- ⚠️ 缺：所有文件的 `type` 字段（目前 0% 有）
- ⚠️ 缺：跨文档的 Markdown 链接（目前是单文件结构）

**KDD 视角**（已有 50% 基础）：
- ✅ K 层：`lessons.jsonl` 是规则库
- ✅ D 数据层：`episodic/` 原始事件
- ⚠️ 决策层弱：当前 `graduate` 是**人工审核**（不是 AI 自主）
- ⚠️ 闭环反馈弱：`memory_reflect.py` 自动聚类但**未触发自动决策**

**5 级成熟度视角**（之前是 0% 基础，现在引入）：
- 🎯 **刚启动**：用这个表给每类资产打分 → 找短板
- 🎯 **每 1-2 月**：重打一次，看升级进度
- 🎯 **设定目标**：以 L5 为长期方向

## 5. 可立刻落地的小改动

按"成本/收益"排序（5 分钟 → 30 分钟）：

### 改动 1：给 LESSONS.md 加 OKF frontmatter（5 分钟）— **⏰ 优先级最高**

```markdown
---
type: lessons-catalog
schema: okf-v0.1-inspired
scope: cross-agent
version: 2026-06-16
tags: [agentic-stack, memory, knowledge-engineering]
---

# Lessons
...（原内容）
```

**收益：** 跨工具（Claude/Codex/Cursor）一眼能识别这是「知识层」资产。
**对应成熟度：** L2 → L3 跨越

### 改动 2：给每条 lesson 加 OKF 字段（10 分钟）

每条 `<!-- id=lesson_xxx -->` 注释行改为 YAML frontmatter：

```markdown
---
type: lesson
id: 2026-06-15-001
conditions: [git, vault]
confidence: 0.485
tags: [ppt, generation]
---
- 任何文件默认存档到 vault + 主动汇报 + Git 同步
```

**收益：** OKF 兼容 + 可被外部 agent 用统一方式消费。
**对应成熟度：** L3 → L4 跨越

### 改动 3：用 5 级成熟度模型给所有 vault 资产打分（30 分钟）

```text
/root/vault/Research/maturity-assessment/
├── index.md                           # type: maturity-report
├── assets/
│   ├── MEMORY.md.md                   # type: asset-score, level: L2
│   ├── daily-logs.md                  # type: asset-score, level: L2
│   ├── wechat-articles.md             # type: asset-score, level: L2
│   ├── research-notes.md              # type: asset-score, level: L2
│   └── lessons-jsonl.md               # type: asset-score, level: L3
└── roadmaps/
    └── 2026-q3-l4-target.md           # type: maturity-roadmap
```

**收益：** 这是「评估层」的真实落地。每月重打一次，看升级进度。
**对应成熟度：** 整个 vault L2 → L3 跨越

### 改动 4：在 `Research/` 下建"OKF 知识库 bundle"（45 分钟）

把今天的 3 份核心资产（OKF 文章、KDD 文章、5 级成熟度文章、本笔记）用 OKF 方式串成 bundle：

```text
/root/vault/Research/okf-bundle/
├── index.md                           # type: bundle-index
├── articles/
│   ├── okf-spec.md                    # type: external-article, source: 谷歌
│   ├── kdd-pyramid.md                 # type: external-article, source: 德勤
│   └── maturity-model.md              # type: external-article, source: 战略流程智慧坊
├── analysis/
│   └── knowledge-triangle.md          # type: research-report（本文件）
└── schemas/
    └── okf-frontmatter.md             # type: schema-definition
```

**收益：** 这就是 OKF 规范的真实 demo 案例。

## 6. 风险与开放问题

### 6.1 OKF 的局限（需观察）

- **v0.1 极简**：仅强制 `type` 字段——可能过简，缺乏 KDD 决策层需要的"决策上下文"字段（如 confidence / conditions / evidence）
- **生产者生态薄弱**：谷歌提供 3 个参考实现，但其它生产者/消费者需自建
- **跨语言检索未解决**：Markdown 适合人读，机器查询需要额外索引层

### 6.2 德勤 KDD 的局限

- **数据可信度低**：原文引用"美的 6 秒节拍/科沃斯 20 倍人效"等数字**未提供方法论和样本**——典型咨询式夸张
- **缺失败案例**：三个都是"标杆"，没有"失败案例"——这是 PPT 推销文常见偏差
- **K 层定义模糊**：「业务规则、SOP」如何数字化？与 OKF 的 frontmatter 怎么对齐？

### 6.3 5 级成熟度的局限

- **「关键 K」的合理性未证**：L3 线上化率≥80%、L4 AI 覆盖率≥30% 等等——是行业惯例还是拍脑袋？
- **升 L5 ≠ 达 L5**：原文说"12-24 月可到 L5"——这是企业 1-2 年就能做到的吗？数据样本不详
- **行业差异被抹平**：制造业 / 金融 / 互联网的成熟度阈值可能完全不同，原文用统一标准

### 6.4 我们的开放问题

- **是否真的需要 OKF 化？** 何大人目前 Markdown 系统已经在跑——加 OKF frontmatter 是**真的简化**还是**复杂度税**？
- **闭环反馈怎么做？** agent 用了 lesson 后怎么知道"用了有没有用"？需要 telemetry 层（类似 KDD 的"实时采集"）
- **小助自主 graduate？** 现在 lessons graduate 是人工——能否让 KDD 决策层做这件事（基于 confidence × evidence 自动升降级）？
- **5 级模型对个人 / 小团队也适用吗？** 原文场景是企业，我们（小助 + 何大人 2 人组）能跳级吗？
