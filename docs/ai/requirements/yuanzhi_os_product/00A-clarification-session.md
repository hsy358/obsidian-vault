## 澄清问题 1（P0-001，最高优先级）

### Data Copilot 的 Text-to-SQL 权限模型

**来源：** US-M01 · 角色权限 · 数据安全

**问题背景：**
校领导通过自然语言向 Data Copilot 提问，后端调用 Text-to-SQL 将自然语言转为 SQL 查询数据库。若无权限控制，大模型可直接拼接任意 SQL（如 `SELECT * FROM teacher_salary`），导致全校敏感数据泄露。这是一个 **安全和合规双重风险** 的 P0 问题。

**为什么必须确认：**
不确认就开发， Coding Agent 可能按最简方案实现（无权限过滤的 Text-to-SQL），上线后即存在数据泄露风险，且后期修复成本极高（涉及架构改动）。

**推荐选项：**

| 选项 | 说明 | 推荐度 |
|---|---|---|
| **A. 预置查询模板（推荐）** | 首版 Text-to-SQL 仅支持预设 SQL 模板列表（如"各专业学生人数"、"本学期开课率"），大模型只负责选择模板和填充参数，不拼接自由 SQL | ⭐⭐⭐ **推荐** |
| B. 基于角色过滤的自由 SQL | 大模型可生成自由 SQL，但查询结果受当前用户角色数据范围过滤（如系部主任只能看本系数据） | 次推荐，但实现复杂度更高 |
| C. 完全自由 SQL + 审计日志 | 不加 SQL 层限制，但记录所有查询日志，事后审计 | 不推荐，安全风险高且合规问题未解决 |
| D. 首版不做 Text-to-SQL，仅做固定报表 | Data Copilot 仅展示预置报表，自然语言交互功能移至二期 | 保守选项，若需快速上线可考虑 |

**推荐答案：A**
**推荐理由：** 安全风险最低，无需新增权限网关，实现最快，适合首版目标（替代 SDS 并上线核心功能）。

**默认假设：** 首版 Text-to-SQL 采用方案 A（预置模板），模板列表由系统管理员在后台配置。

**假设错误影响：** 若实际需要自由 SQL 功能（方案 B），需新增权限网关、查询审计和防注入机制，预计增加 2 周开发周期。

**建议确认话术：**
> "Data Copilot 的 Text-to-SQL 能力，首版计划采用预设 SQL 模板方案，不开放自由 SQL 生成以避免数据泄露风险。查询结果按当前用户角色数据范围过滤。这个方案是否满足当前需求？如需自由 SQL，建议放在二期并单独做安全评审。"

## 澄清问题 2（P0-002，高优先级）

### AI 任务裂变的粒度定义与状态机

**来源：** US-P02 · 业务对象 · 状态流转

**问题背景：**
"点击 AI 分解，长文自动裂变为几十张可追踪的微任务卡片"——这是专业空间最核心的 AI 功能之一。但裂变的粒度（按什么单位拆分）决定：任务卡片的 Schema 设计、状态机设计、前端渲染逻辑和任务分配逻辑。

**为什么必须确认：**
粒度不明确的情况下， Coding Agent 只能用推测实现，可能导致：卡片信息太碎片化（按句子拆分）或信息太粗糙（按章节拆分），无法支持实际任务追踪。

**推荐选项：**

| 选项 | 说明 | 推荐度 |
|---|---|---|
| **A. 按语义段落拆分（推荐）** | AI 分析长文语义，将完整段落拆分为独立任务卡片，每张卡片包含：任务描述、原始段落引用、负责人（默认空）、截止日期（默认空） | ⭐⭐⭐ **推荐** |
| B. 按行动计划项拆分 | AI 提取长文中的行动计划项（以动词开头），每项生成一张卡片，颗粒度更细 | 实现复杂度更高，适合精细化任务管理场景 |
| C. 按年度目标拆分为季度/月度任务 | AI 理解年度规划，自动拆解为季度/月度里程碑卡片 | 适合规划类文档，但范围较窄 |
| D. 首版先做手工任务创建，AI 拆分配套二期 | 手动创建任务卡片，AI 仅提供建议拆分方案（人工确认后生成） | 保守选项，风险最低 |

**推荐答案：A**
**推荐理由：** 语义段落是文档自然结构，拆分结果可读性强，实现复杂度适中，首版可快速验证。

**默认假设：** AI 按语义段落裂变，每段一张微任务卡片；卡片状态机：待领取 → 进行中 → 已完成 → 已验收（4态）。

**假设错误影响：** 若实际需要更细粒度（方案 B），需重新设计卡片 Schema，增加"行动项类型"字段；若需要层级结构（方案 C），需新增任务层级关系。

**建议确认话术：**
> "AI 分解功能，首版计划按语义段落自动生成任务卡片，每段一张，包含原文引用，支持手动调整负责人和截止日期。卡片状态流转：待领取→进行中→已完成→已验收。是否满足专业空间的任务追踪需求？如需更细粒度（如行动计划项），建议在卡片 Schema 中预留扩展字段。"

## 澄清问题 3（P0-003，高优先级）

### 表单拖拽配置器的配置边界

**来源：** US-M03 · 业务边界 · 系统开放性

**问题背景：**
"系统管理员通过拖拽式设计器配置表单和审批流程"——这是管理空间最灵活的功能，也是最容易过度设计的功能。若边界不清，可能：做得太窄（用户不认可）或做得太宽（开发周期不可控）。

**为什么必须确认：**
这是直接影响首版工期判断和研发排期的 P0 问题。不同配置边界，开发工作量可能差 3-5 倍。

**推荐选项：**

| 选项 | 说明 | 推荐度 |
|---|---|---|
| **A. 轻量配置（推荐）** | 表单：字段顺序调整、必填/非必填切换、字段显隐；审批：单节点审批（单人审批通过）；不支持自定义字段类型、公式、外部数据源 | ⭐⭐⭐ **推荐** |
| B. 中等配置 | 在 A 基础上，支持多节点审批（含会签/或签）、字段公式计算（简单加减乘除）、下拉联动 | 次推荐，但 BPMN 实现复杂度显著上升 |
| C. 完整 BPMN | 支持完整 BPMN 2.0流程（多实例、并行网关、计时器、脚本节点等） | 不推荐，首版工期不可控 |
| D. 首版不做表单设计器，提供固定表单模板 | 表单内容通过配置文件管理，管理员无权自行设计 | 保守选项，功能受限但稳定 |

**推荐答案：A**
**推荐理由：** 轻量配置可满足首版核心需求（底层配置底座），开发周期可控，且预留了未来扩展至 B 的接口。

**默认假设：** 表单设计器首版仅支持字段顺序、必填/非必填、显隐配置；审批设计器仅支持单节点审批（单人）；不提供公式编辑器和外部数据源绑定。

**假设错误影响：** 若实际需要方案 B 或 C，需重新评估工期，增加 3-8 周开发周期。建议二期迭代。

**建议确认话术：**
> "表单设计器首版采用轻量方案：仅支持字段顺序调整、必填切换和单节点审批配置。审批流程暂不支持多节点和会签/或签。这个范围是否满足管理空间的最小可用需求？如需完整 BPMN，建议在架构设计中预留引擎接口，二期再做。"

## 澄清问题 4（P0-004，中高优先级）

### 评审专家在专业空间的具体权限

**来源：** 第二章角色权限映射表 · 角色权限 · 数据安全

**问题背景：**
评审专家在专业空间有"核心视图"，但具体能做什么未定义。核心视图是否包含编辑权限？能否驳回/退回？能否查看草稿状态数据？若权限过宽，评审专家可能修改正式数据；若权限过窄，可能无法完成评审职责。

**推荐选项：**

| 选项 | 说明 | 推荐度 |
|---|---|---|
| **A. 只读（含历史版本）+ 评审意见（推荐）** | 评审专家可查看所有正式版本和历史版本，可提交评审意见（文本），不可编辑、删除、审批或驳回 | ⭐⭐⭐ **推荐** |
| B. 只读 + 可驳回 | 在 A 基础上，评审专家可将待审内容驳回至草稿状态，附驳回原因 | 次推荐，需新增驳回状态和原因字段 |
| C. 只读 + 可审批通过 | 在 A 基础上，评审专家可一键审批通过（无需文字意见） | 实现较简单，但可能过于草率 |
| D. 首版不给评审专家开账号 | 评审专家功能移至二期，首版仅支持内部评审 | 最保守选项 |

**推荐答案：A**
**推荐理由：** 评审专家的职责是提供专业意见，只读+意见模式已可满足核心需求，且实现成本最低，风险最小。

**默认假设：** 评审专家在专业空间只有只读权限（含历史版本）和提交评审意见功能；不提供编辑/审批/驳回/删除权限。

**假设错误影响：** 若实际需要驳回功能（方案 B），需新增：评审意见表（含驳回原因字段）、状态驳回动作、驳回通知和日志。

**建议确认话术：**
> "评审专家在专业空间，首版计划开放：只读所有正式版本和历史版本 + 提交评审意见（文本）。不提供编辑、审批、驳回权限。是否满足评审专家的评审职责？如需驳回功能，建议在评审意见表中增加'驳回'类型字段，并在状态机中增加驳回状态。"
