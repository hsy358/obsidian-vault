## 2. 详细隐含用例推导

### UC-I01：SDS 数据迁移

**推导来源**：需求原文"目标是全面替代传统 SDS 教学管理系统"

**证据**：
- 需求明确"全面替代"
- 涉及校领导数据看板（需迁移历史数据）
- 涉及专业建设数据（需迁移专业、课程、用户关联）

**隐含内容**：
- 迁移粒度（专业/课程/用户/资源）
- 迁移方式（ETL 工具/手动导入/增量同步）
- 迁移后 SDS系统的处理（停用/归档/只读）
- 历史数据格式转换（SDS 结构 → Yuanzhi OS 结构）

**缺失能力**：
- `data.migration` 能力未在 CAPABILITY-GRAPH.yaml 中体现
- `storage.postgres` 中是否已有迁移相关脚本未知

**阻塞问题**：
- Q014: SDS 数据迁移的粒度、方式、时间窗口是什么？是否需要双系统并行运行过渡期？

---

### UC-I02：AI 任务取消与重试

**推导来源**：AI 任务状态机（pending→running→completed/failed）

**证据**：
- UC-004/UC-005 涉及异步 AI 任务
- 状态机中 failed状态可重试
- 异常流程 E1（任务失败）提到"可一键重试"

**隐含内容**：
- running 状态下是否允许取消？（部分 AI 服务支持中途取消）
- 重试是否保留原输入？还是重新发起？
- 重试次数是否有限制？
- 重试失败后的处理策略

**工程映射**：
- 前端：AISidebar.vue 需展示"取消"和"重试"按钮
- 后端：AiChatService.java 需实现 cancelMission() 和 retryMission() 方法
- 数据：ai_mission 表需增加 retry_count 和 cancelled_at 字段

**缺失能力**：
- `ai.mission-cancel` 和 `ai.mission-retry` 能力未在 CAPABILITY-GRAPH.yaml 中体现

**阻塞问题**：
- Q015: AI 任务重试策略是什么？最多重试几次？重试间隔？

---

### UC-I03：通知催办机制

**推导来源**：BR-005"评审时限7 个工作日，超时自动催办"

**证据**：
- 评审专家 UC-007 分支 G2 提到"评审超时自动催办"
- 业务规则明确催办机制

**隐含内容**：
- 催办通知的发送方式（站内消息/邮件/短信）
- 超时判断（7个工作日不含周末？节假日是否顺延？）
- 催办次数限制（超多少次后升级通知？）
- 催办任务的调度实现（定时任务/Cron）

**工程映射**：
- 后端：需实现定时任务调度（如 Spring @Scheduled 或 Quartz）
- 通知服务：需接入通知系统（站内消息/邮件）
- 数据库：`notification` 表设计

**缺失能力**：
- `notification.system` 未在 CAPABILITY-GRAPH.yaml 的 existing 中体现（仅在 missing_capabilities）

**阻塞问题**：
- Q016: 催办通知的发送渠道是什么？站内消息、邮件还是短信？超时判断是否含节假日？

---

### UC-I04：权限变更后的会话处理

**推导来源**：UC-013 权限配置"权限变更实时生效（已登录用户需重新登录）"

**证据**：
- 业务规则要求权限变更实时生效
- 但已登录用户的会话状态如何处理未明确

**隐含内容**：
- 权限变更后是否强制当前在线用户重新登录？
- 还是仅禁止后续操作，已在线用户可继续当前会话？
- 是否需要刷新 token 机制？

**工程映射**：
- 前端：store 需支持权限刷新
- 后端：AuthService.java 需实现权限刷新或会话失效机制
- Token 策略：JWT 的 short-lived token + refresh token

**假设**：
- 权限变更后，已登录用户需重新登录方可获得新权限（默认方案）

### UC-I05：资源版本管理

**推导来源**：UC-001 分支 A3"文件已存在（MD5 重复）"

**证据**：
- 提出了文件 MD5 去重的场景
- 但未说明是否支持多版本

**隐含内容**：
- 是否支持同一资源的多版本（如课件更新）？
- 版本策略：是覆盖还是新增版本号？
- 历史版本是否可查看/恢复？
- 版本列表是否展示在资源详情页？

**工程映射**：
- 数据库：`resource` 表需增加 version字段和 parent_resource_id
- 前端：资源详情页需展示版本历史

**缺失能力**：
- `resource.versioning` 未在 CAPABILITY-GRAPH.yaml 中体现

### UC-I06：图谱节点冲突处理

**推导来源**：UC-006 分支 F1"已有图谱 → 提示覆盖或追加"

**证据**：
- 提出了图谱构建时可能覆盖已有图谱的场景

**隐含内容**：
- 冲突检测粒度（节点级别？关系级别？）
- 冲突时的默认策略（默认不覆盖？）
- 增量追加时如何处理重复节点（合并/跳过/新建）？
- 图谱变更是否需要审计日志？

**工程映射**：
- 后端：GraphitiService.java 需实现冲突检测逻辑
- 数据库：图谱变更记录写入 audit_log

**缺失能力**：
- `kg.conflict-detection` 未在 CAPABILITY-GRAPH.yaml 中体现

### UC-I07：跨专业共享课程

**推导来源**：知识图谱构建时课程可能跨专业关联

**证据**：
- 课程知识图谱可能涉及多个专业共用的基础课程
- 权限矩阵中课程设计属于专业带头人（R04）

**隐含内容**：
- 一个课程节点是否可以关联多个专业？
- 跨专业编辑课程时是否需要审批？
- 专业带头人只能编辑自己专业的课程图谱

**工程映射**：
- 数据库：`knowledge_node` 需支持多专业关联（many-to-many）
- 权限：专业带头人的图谱编辑权限需限定在专业范围内

**缺失能力**：
- `kg.cross-professional` 未在 CAPABILITY-GRAPH.yaml 中体现

### UC-I08：AI 响应内容合规审查

**推导来源**：UC-012 AI 辅助教案"内容违规 → 拒绝输出"

**证据**：
- AI 生成内容可能涉及敏感词、政治问题等
- 合规要求必须有内容审核机制

**隐含内容**：
- 审核是前置（生成前）还是后置（生成后）？
- 审核粒度（关键词过滤/语义审核/人工抽检）？
- 违规内容是否记录日志并通知管理员？

**工程映射**：
- 后端：需接入内容审核服务（如阿里云内容安全/自建敏感词库）
- AiChatService.java 需在返回前调用审核服务

**缺失能力**：
- `ai.content-moderation` 未在 CAPABILITY-GRAPH.yaml 中体现

**阻塞问题**：
- Q017: AI 内容审核机制是什么？是实时过滤还是后置人工抽检？审核失败是否阻断输出？

### UC-I09：培养方案版本管理

**推导来源**：状态机 published→drafting（重新编辑需重新审核）

**证据**：
- 方案发布后如果重新编辑，会触发新的审核流程
- 但未说明历史版本如何处理

**隐含内容**：
- 每次提交审核是否自动生成版本号？
- 历史版本是否可查看和对比？
- 是否支持版本回滚？

**工程映射**：
- 数据库：`program_plan` 表需支持多版本（version_id 或 history 表）
- 前端：方案详情页需展示版本历史

**缺失能力**：
- `plan.version-history` 未在 CAPABILITY-GRAPH.yaml 中体现
